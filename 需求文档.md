## 1. 项目概述

### 1.1 项目定位

`sdd-harness` 是一个面向 **Claude Code** 和 **Codex** 的代码库感知型 SDD Harness 插件框架。

它的目标是在 Claude Code / Codex 之上增加一层受控开发流程，使 AI 编码从自由执行变成可约束、可验证、可追踪的工程流程。

核心流程：

```text
初始化项目
  ↓
代码库索引
  ↓
需求澄清
  ↓
规格生成
  ↓
方案设计
  ↓
任务拆解
  ↓
实现
  ↓
功能验证
  ↓
代码审查
  ↓
归档
```

### 1.2 一句话描述

```text
sdd-harness 是一个运行在 Claude Code 和 Codex 之上的代码库感知型 SDD Harness 插件，通过 codebase-memory-mcp、内置 Spec Engine、内置 TDD Engine、阶段化命令和 .sdd 事实源，让 AI 编码流程可控、可验证、可追踪。
```

## 2. 交付形态

### 2.1 第一版交付形态

第一版按 **Claude Code / Codex 插件包** 交付，暂不实现独立 CLI。

也就是说，第一版重点支持：

```text
1. Claude Code 插件包
2. Claude Code slash commands
3. Claude Code skills
4. Codex 项目指令
5. Codex skills
6. AGENTS.md
7. CLAUDE.md
```

暂不发布：

```text
1. npm 全局 CLI
2. Homebrew 包
3. 独立二进制
4. Web UI
5. 后台服务
```

### 2.2 命令形态说明

虽然文档中统一写作：

```text
sdd init
sdd auto
sdd new
...
```

但第一版中它们是插件命令契约，不是独立 CLI。

Claude Code 中表现为：

```text
/sdd.init
/sdd.auto "粗略需求"
/sdd.new "粗略需求"
/sdd.design
/sdd.plan
/sdd.build
/sdd.verify
/sdd.review
/sdd.archive
/sdd.status
```

Codex 中表现为：

```text
sdd init
sdd auto "粗略需求"
sdd new "粗略需求"
sdd design
sdd plan
sdd build
sdd verify
sdd review
sdd archive
sdd status
```

### 2.3 为什么仍定义命令契约

即使第一版暂不实现独立 CLI，也必须定义统一命令契约，原因是：

```text
1. Claude Code slash command 需要清晰输入输出
2. Codex skill 需要清晰执行协议
3. 后续独立 CLI 可以复用同一套契约
4. 自动测试可以按命令契约验证
5. sdd auto 需要内部编排这些命令
```

### 2.4 共享可执行核心

第一版虽然不发布独立 CLI，但必须随插件分发一个共享的确定性执行核心：

```text
Claude Code slash command ─┐
                           ├→ Plugin Adapter → @sdd-harness/core → Filesystem / Git / MCP
Codex skill ────────────────┘
```

职责边界：

```text
1. Claude Code Adapter 负责解析 slash command、展示交互和映射 Core 结果
2. Codex Adapter 负责解析 skill 输入、展示交互和映射 Core 结果
3. @sdd-harness/core 负责状态机、Schema 校验、文件锁、原子写入、Git 检查、制品管理和 MCP 调用
4. SpecEngine 和 TddEngine 作为 Core 内部模块运行
5. Adapter 不得自行实现或绕过状态流转和安全检查
6. 两个平台必须对相同输入返回一致的状态、错误码和制品
```

第一版技术基线：

```text
1. TypeScript 5.x
2. Node.js 20 LTS 或更高 LTS 版本
3. Core 以项目本地依赖随插件分发，不要求用户全局安装 CLI
4. 插件命令通过稳定的 Core API 调用，不通过模拟终端命令调用
5. 发布时必须在 package.json engines 中声明 Node.js 版本范围
```

Core 最小调用契约：

```typescript
export interface SddCore {
  execute(request: CommandRequest): Promise<CommandResult>;
}

export interface CommandRequest {
  command: "init" | "auto" | "new" | "design" | "plan" | "build" | "verify" | "review" | "archive" | "status";
  cwd: string;
  args: Record<string, unknown>;
  signal?: AbortSignal;
}

export interface CommandResult {
  ok: boolean;
  state: string;
  exitCode: number;
  error?: { code: string; message: string; next?: string };
}
```

文档中的“退出码”在插件模式下表示 `CommandResult.exitCode`；未来独立 CLI 直接将该值映射为进程退出码。

## 3. 核心设计原则

### 3.1 `.sdd/` 是唯一事实源

所有最终制品、状态、上下文、报告和归档记录都必须放在：

```text
.sdd/
```

禁止将 OpenSpec、Superpowers 或其他工具的默认目录作为最终事实源。

### 3.2 codebase-memory-mcp 使用原版

代码库索引和代码图谱能力不自研，直接使用原版：

```text
codebase-memory-mcp
```

该组件用于减少 Agent 反复文件读取和 grep 的 token 消耗；公开论文描述 Codebase-Memory 通过 Tree-Sitter 构建持久代码图谱，并提供调用图、影响分析等能力，报告在评测中使用更少工具调用和显著更低 token 成本。

### 3.3 OpenSpec 能力内置化

OpenSpec 不作为外部运行时工具直接调用。

`sdd-harness` 应在许可证允许后，复制 / fork / vendor OpenSpec 的相关代码和思想，改造成内部模块：

```text
SpecEngine
```

SpecEngine 输出必须写入 `.sdd/changes/<change-id>/`。

### 3.4 Superpowers 能力内置化

Superpowers 不作为外部运行时工具直接调用。

`sdd-harness` 应在许可证允许后，复制 / fork / vendor Superpowers 的相关代码和思想，改造成内部模块：

```text
TddEngine
```

TddEngine 输出必须写入 `.sdd/changes/<change-id>/`。

### 3.5 许可证先行

OpenSpec 和 Superpowers 的复制、fork、vendor 不能在许可证确认前作为不可逆实现。

实现前必须完成：

```text
1. 确认 LICENSE
2. 确认是否允许复制
3. 确认是否允许修改
4. 确认是否允许再分发
5. 确认是否允许商业或内部使用
6. 保留 LICENSE
7. 保留 copyright notice
8. 生成 THIRD_PARTY_NOTICES.md
```

如果许可证不允许复制或内置改造，则只能重新实现相同思想，不能直接搬代码。

## 4. 外部组件契约

### 4.1 codebase-memory-mcp 契约

#### 4.1.1 使用策略

`codebase-memory-mcp` 使用原版，不复制、不 fork、不重写。

#### 4.1.2 init 检查行为

执行 `sdd init` 时必须检查：

```text
1. codebase-memory-mcp 是否已安装
2. MCP 配置是否存在
3. Claude Code 或 Codex 是否能连接该 MCP
4. MCP 服务是否可调用
5. 当前仓库是否已被索引
```

#### 4.1.3 未安装行为

如果未安装，系统不得直接失败退出，而是：

```text
1. 给出安装建议
2. 给出官方安装入口或项目说明
3. 标记 codebase provider 为 unavailable
4. 降级为普通文件查找和阅读模式
5. 在 status 中提示当前处于 degraded mode
```

#### 4.1.4 不可用降级策略

当 codebase-memory-mcp 不可用时，降级为：

```text
1. 文件名搜索
2. 目录结构扫描
3. 关键字 grep
4. 读取候选文件
5. 根据包结构生成简化 codebase-summary.md
```

降级模式必须设置：

```json
{
  "codebaseProvider": "fallback-file-scan",
  "degraded": true,
  "reason": "codebase-memory-mcp unavailable"
}
```

#### 4.1.5 支持版本和接口

MVP 固定使用 `codebase-memory-mcp v0.8.1`，对应完整 commit `f0c9be19c5d74b84f418d807bfdce7b5d6a261ff`，不得在构建时动态使用 latest。

固定信息必须写入：

```text
.sdd/adapters/codebase-memory-mcp/version.json
```

示例：

```json
{
  "name": "codebase-memory-mcp",
  "version": "v0.8.1",
  "commit": "f0c9be19c5d74b84f418d807bfdce7b5d6a261ff",
  "checksumManifest": "https://github.com/DeusData/codebase-memory-mcp/releases/download/v0.8.1/checksums.txt",
  "checksumManifestSha256": "142399e4e552fb559ede866b2549dbacc942d56f1c8718b52bc701b21f3f94c6",
  "resolvedAt": "2026-07-03T00:00:00+08:00",
  "interface": "mcp",
  "status": "available"
}
```

macOS 和 Windows 安装器必须先验证 `checksums.txt` 自身摘要，再使用其中对应平台条目验证下载产物。任一摘要不匹配时返回 `E_COMPONENT_INTEGRITY_FAILED`，不得启动该组件。

### 4.2 OpenSpec 契约

MVP 固定使用 OpenSpec `v1.4.1`，对应完整 commit `1b06fddd59d8e592d5b5794a1970b22867e85b1f`。未完成许可证确认和版本固定前，不得复制、fork 或 vendor 其代码。

必须记录：

```text
.sdd/adapters/openspec/version.json
vendor/openspec/LICENSE
```

记录内容必须包含来源仓库、版本、完整 commit、许可证和本地修改说明。Git 源码以完整 commit SHA 作为内容完整性标识；如果改用下载归档，还必须记录归档 SHA-256。后续同步升级必须显式发起，并通过 SpecEngine 单元测试、契约测试和回归测试；不得自动跟随上游最新版本。

### 4.3 Superpowers 契约

MVP 固定使用 Superpowers `v6.1.1`，对应完整 commit `d884ae04edebef577e82ff7c4e143debd0bbec99`。未完成许可证确认和版本固定前，不得复制、fork 或 vendor 其代码。

必须记录：

```text
.sdd/adapters/superpowers/version.json
vendor/superpowers/LICENSE
```

记录内容必须包含来源仓库、版本、完整 commit、许可证和本地修改说明。Git 源码以完整 commit SHA 作为内容完整性标识；如果改用下载归档，还必须记录归档 SHA-256。后续同步升级必须显式发起，并通过 TddEngine 单元测试、契约测试和回归测试；不得自动跟随上游最新版本。

## 7. 命令契约

### 7.1 通用参数

第一版不发布独立 CLI，但所有插件命令必须按以下契约设计，方便后续迁移为 CLI。

通用选项：

| 选项                    |      类型 |     默认值 | 说明                 |
| --------------------- | ------: | ------: | ------------------ |
| `--json`              | boolean |   false | 输出机器可读 JSON        |
| `--non-interactive`   | boolean |   false | 禁止交互提问             |
| `--force`             | boolean |   false | 强制覆盖候选制品，MVP0 可不实现 |
| `--timeout <seconds>` |  number |       0 | 0 表示不主动超时          |
| `--change <id>`       |  string | current | 指定 Change          |
| `--verbose`           | boolean |   false | 输出调试信息             |

### 7.2 `--help`

每个命令必须支持 help 文档。

示例：

```text
/sdd.status --help
sdd status --help
```

输出必须包含：

```text
1. 命令用途
2. 参数列表
3. 默认值
4. 示例
5. 可能退出码
```

### 7.3 `--version`

插件包必须提供版本信息。

示例输出：

```json
{
  "name": "sdd-harness",
  "version": "0.1.0",
  "delivery": "plugin",
  "supportedTargets": ["claude-code", "codex"]
}
```

### 7.4 标准输出格式

人类可读输出：

```text
SDD Status: PLAN_READY

Change:
add-order-cancel

Next:
sdd build
```

机器可读输出：

```json
{
  "ok": true,
  "state": "PLAN_READY",
  "changeId": "add-order-cancel",
  "next": "sdd build"
}
```

### 7.5 标准错误格式

人类可读错误：

```text
Error: E_INVALID_PHASE_COMMAND
Cannot run build before design and plan are completed.
Next: sdd design
```

JSON 错误：

```json
{
  "ok": false,
  "error": {
    "code": "E_INVALID_PHASE_COMMAND",
    "message": "Cannot run build before design and plan are completed.",
    "next": "sdd design"
  }
}
```

### 7.6 退出码

|   退出码 | 含义          |
| ----: | ----------- |
|   `0` | 成功          |
|   `1` | 通用失败        |
|   `2` | 参数错误        |
|   `3` | 状态非法        |
|   `4` | 缺少制品        |
|   `5` | 外部组件不可用     |
|   `6` | 用户输入阻塞      |
|   `7` | 验证失败        |
|   `8` | 审查失败        |
|   `9` | 并发锁失败       |
|  `10` | 安全策略阻断      |
| `130` | 用户中断 Ctrl+C |
| `124` | 超时          |

### 7.7 交互模式

默认允许交互。

`sdd new` 和 `sdd auto` 遇到 BLOCKER 问题时必须暂停，保存状态为：

```text
CLARIFYING
```

并写入：

```text
.sdd/changes/<change-id>/questions.md
.sdd/state.json
```

用户回答后继续当前阶段。

### 7.8 非交互模式

如果传入：

```text
--non-interactive
```

遇到 BLOCKER 时必须失败，不得猜测。

返回：

```text
E_UNRESOLVED_BLOCKER
```

退出码：

```text
6
```

### 7.9 Ctrl+C 行为

用户中断时：

```text
1. 尽可能保存当前 state.json
2. 写入 audit.log
3. 释放 lock
4. 当前阶段固定标记为 PAUSED
5. 记录 interruptedCommand、previousPhase 和 suggestedCommand
6. 返回退出码 130
```

### 7.10 超时行为

如果设置 `--timeout`，超时后：

```text
1. 停止当前阶段
2. 保存状态
3. 释放 lock
4. 写入 audit.log
5. 当前阶段固定标记为 FAILED
6. 记录错误码 E_TIMEOUT、failedCommand、previousPhase 和 suggestedCommand
7. 返回退出码 124
```

## 8. 状态机设计

### 8.1 状态枚举

```text
NOT_INITIALIZED
INITIALIZING
INDEXING
INDEX_READY
NEW_STARTED
CLARIFYING
SPEC_READY
DESIGNING
DESIGN_READY
PLANNING
PLAN_READY
BUILDING
BUILD_READY
VERIFYING
VERIFY_READY
REVIEWING
REVIEW_READY
ARCHIVING
ARCHIVED
FAILED
PAUSED
```

### 8.2 状态含义

| 状态                | 含义                        |
| ----------------- | ------------------------- |
| `NOT_INITIALIZED` | 当前项目未初始化                  |
| `INITIALIZING`    | 正在初始化                     |
| `INDEXING`        | 正在索引代码库                   |
| `INDEX_READY`     | 初始化完成，索引可用                |
| `NEW_STARTED`     | 已创建 Change，正在处理需求         |
| `CLARIFYING`      | 等待用户回答 BLOCKER 问题         |
| `SPEC_READY`      | spec 已完成                  |
| `DESIGNING`       | 正在生成和校验 design            |
| `DESIGN_READY`    | design 已完成                |
| `PLANNING`        | 正在生成和校验 tasks 与 context packs |
| `PLAN_READY`      | tasks 和 context packs 已完成 |
| `BUILDING`        | 正在实现                      |
| `BUILD_READY`     | 实现完成                      |
| `VERIFYING`       | 正在验证                      |
| `VERIFY_READY`    | 验证通过                      |
| `REVIEWING`       | 正在审查                      |
| `REVIEW_READY`    | 审查通过                      |
| `ARCHIVING`       | 正在归档                      |
| `ARCHIVED`        | 已归档，只读                    |
| `FAILED`          | 失败，等待重新执行建议命令             |
| `PAUSED`          | 被用户中断或主动暂停                |

## 9. 状态 × 命令矩阵

### 9.1 主流转矩阵

| 当前状态              | 命令              | 下一状态                                    | 结果                       |
| ----------------- | --------------- | --------------------------------------- | ------------------------ |
| `NOT_INITIALIZED` | `sdd init`      | `INITIALIZING → INDEXING → INDEX_READY` | 成功                       |
| `NOT_INITIALIZED` | `sdd status`    | `NOT_INITIALIZED`                       | 成功                       |
| `NOT_INITIALIZED` | 其他命令            | 不变                                      | `E_NOT_INITIALIZED`      |
| `INDEX_READY`     | `sdd new "需求"`  | `NEW_STARTED → CLARIFYING 或 SPEC_READY` | 成功或等待回答                  |
| `INDEX_READY`     | `sdd auto "需求"` | 自动流转                                    | 成功或中断                    |
| `INDEX_READY`     | `sdd status`    | `INDEX_READY`                           | 成功                       |
| `INDEX_READY`     | `sdd init`      | `INDEX_READY`                           | 幂等检查                     |
| `CLARIFYING`      | `sdd new`       | `NEW_STARTED → SPEC_READY`              | 读取回答并继续生成 spec          |
| `CLARIFYING`      | `sdd status`    | `CLARIFYING`                            | 成功                       |
| `CLARIFYING`      | 其他写命令           | 不变                                      | `E_UNRESOLVED_BLOCKER`     |
| `SPEC_READY`      | `sdd design`    | `DESIGNING → DESIGN_READY`              | 成功                       |
| `SPEC_READY`      | `sdd auto`      | 从 design 继续                             | 成功或中断                    |
| `SPEC_READY`      | `sdd status`    | `SPEC_READY`                            | 成功                       |
| `SPEC_READY`      | `sdd new`       | 不变                                      | `E_ACTIVE_CHANGE_EXISTS` |
| `DESIGN_READY`    | `sdd plan`      | `PLANNING → PLAN_READY`                 | 成功                       |
| `DESIGN_READY`    | `sdd auto`      | 从 plan 继续                               | 成功或中断                    |

## 11. FAILED 和 PAUSED 恢复规则

### 11.1 失败上下文

失败时必须写入：

```json
{
  "currentPhase": "FAILED",
  "previousPhase": "PLAN_READY",
  "inProgressPhase": "BUILDING",
  "failedCommand": "sdd build",
  "failedReason": "TASK-002 test failed",
  "recoverable": true,
  "suggestedCommand": "sdd build"
}
```

### 11.2 恢复规则

恢复命令必须以 `failedCommand` 或 `interruptedCommand` 为准，不能仅根据状态推断。`previousPhase` 保存命令开始前的最后稳定状态，`inProgressPhase` 保存中断时的执行状态；二者只用于验证命令和恢复失败时回滚。

| 失败或中断命令       | 合法 previousPhase / inProgressPhase               | 建议命令          | 成功后状态          |
| ------------- | -------------------------------------------- | ------------- | -------------- |
| `sdd init`    | `NOT_INITIALIZED`、`INITIALIZING`、`INDEXING`    | `sdd init`    | `INDEX_READY`  |
| `sdd new`     | `INDEX_READY`、`NEW_STARTED`、`CLARIFYING`       | `sdd new`     | `SPEC_READY`   |
| `sdd design`  | `SPEC_READY`、`DESIGNING`                     | `sdd design`  | `DESIGN_READY` |
| `sdd plan`    | `DESIGN_READY`、`PLANNING`                    | `sdd plan`    | `PLAN_READY`   |
| `sdd build`   | `PLAN_READY`、`BUILDING`                      | `sdd build`   | `BUILD_READY`  |
| `sdd verify`  | `BUILD_READY`、`VERIFYING`                     | `sdd verify`  | `VERIFY_READY` |
| `sdd review`  | `VERIFY_READY`、`REVIEWING`                    | `sdd review`  | `REVIEW_READY` |
| `sdd archive` | `REVIEW_READY`、`ARCHIVING`                    | `sdd archive` | `ARCHIVED`     |
| `sdd auto`    | 任一未归档合法状态                                   | `sdd auto`    | 从原阶段继续         |

如果记录的命令与状态组合不匹配，返回 `E_STATE_CORRUPTED`，不得猜测或自动跨阶段执行。进程异常退出导致状态仍停留在 `DESIGNING`、`PLANNING` 等执行中状态时，下一次写命令必须先根据 lock、audit log 和状态映射将其规范化为 `FAILED`，并补全失败上下文。

状态用途固定如下：

```text
CLARIFYING = 等待 BLOCKER 回答
PAUSED     = 用户主动中断
FAILED     = 超时、外部组件错误、校验失败或执行异常
```

### 11.3 不提供 resume 命令

MVP 不提供 `sdd resume`。

恢复方式：

```text
1. sdd status
2. 查看 suggestedCommand
3. 执行 suggestedCommand
```

## 12. 幂等规则

### 12.1 总体规则

所有命令必须安全重复执行。

不得出现：

```text
1. 重复创建 Change
2. 静默覆盖人工修改
3. 重复归档
4. 丢失 task 状态
5. 破坏 state.json
6. 破坏 audit.log
```

### 12.2 命令幂等表

| 命令            | 已完成状态          | 重复执行行为                                        |
| ------------- | -------------- | --------------------------------------------- |
| `sdd init`    | `INDEX_READY`  | 检查并补齐缺失配置，不覆盖已有配置                             |
| `sdd new`     | `SPEC_READY`   | 返回 active change exists                       |
| `sdd design`  | `DESIGN_READY` | 若输入 hash 未变，返回 already ready；若变更，生成 candidate |
| `sdd plan`    | `PLAN_READY`   | 若输入 hash 未变，返回 already ready；若变更，生成 candidate |
| `sdd build`   | `BUILD_READY`  | 只执行 FAILED / PENDING task，不重复 DONE task       |
| `sdd verify`  | `VERIFY_READY` | git diff 未变则返回上次结果，否则重新 verify                |
| `sdd review`  | `REVIEW_READY` | verify 和 git diff 未变则返回上次结果，否则重新 review       |
| `sdd archive` | `ARCHIVED`     | 返回 already archived                           |
| `sdd auto`    | 中间状态           | 从当前状态继续                                       |
| `sdd status`  | 任意状态           | 只读                                            |

### 12.3 文档覆盖规则

如果目标文档存在且检测到人工修改，不得静默覆盖。

应生成：

```text
<filename>.candidate.md
```

并提示用户人工合并。

## 13. state.json 原子写入和恢复

### 13.1 原子写入

禁止直接覆盖 `.sdd/state.json`。

必须执行：

```text
1. 读取 state.json
2. 校验 schema
3. 写 state.json.tmp
4. fsync
5. state.json → state.json.bak
6. rename state.json.tmp → state.json
7. 写 audit.log
```

### 13.2 version 字段

`state.json` 必须包含：

```json
{
  "schemaVersion": "1.0.0",
  "version": 1,
  "updatedAt": "2026-07-03T10:00:00+08:00"
}
```

每次更新 `version + 1`。

### 13.3 损坏判定

以下视为损坏：

```text
1. JSON 解析失败
2. schema 校验失败
3. currentPhase 不合法
4. currentChangeId 指向不存在目录
5. version 缺失
```

### 13.4 恢复流程

```text
1. 尝试恢复 state.json.bak
2. 如果 bak 无效，扫描 .sdd/changes/
3. 根据制品完整性推断状态
4. 生成 state.recovered.json
5. status 中提示用户确认
```

### 13.5 状态推断规则

| 制品情况                                          | 推断状态                              |
| --------------------------------------------- | --------------------------------- |
| 无 `.sdd/`                                     | `NOT_INITIALIZED`                 |
| 有 `.sdd/`，无索引                                 | `NOT_INITIALIZED` 或 `INDEXING` 失败 |
| 有索引，无 active Change                           | `INDEX_READY`                     |
| 有 `spec.md`，无 `design.md`                     | `SPEC_READY`                      |
| 有 `design.md`，无 `tasks.md`                    | `DESIGN_READY`                    |
| 有 `tasks.md`，任务未完成                            | `PLAN_READY`                      |
| 任务全部完成，无 `verify-report.md`                   | `BUILD_READY`                     |
| `verify-report.md` PASS，无 `review-report.md`  | `VERIFY_READY`                    |
| `review-report.md` PASS，无 `archive-report.md` | `REVIEW_READY`                    |
| 有 `.archived`                                 | `ARCHIVED`                        |

## 14. 并发控制

### 14.1 全局锁

所有写命令必须获取：

```text
.sdd/lock
```

只读命令 `sdd status` 不需要写锁。

### 14.2 锁文件

```json
{
  "pid": 12345,
  "command": "sdd build",
  "changeId": "add-order-cancel",
  "createdAt": "2026-07-03T10:00:00+08:00",
  "expiresAt": "2026-07-03T10:10:00+08:00"
}
```

### 14.3 锁规则

```text
1. lock 不存在时创建
2. lock 存在且未过期时返回 E_CONCURRENT_RUN
3. lock 存在且过期时检查 pid
4. pid 不存在则抢占
5. pid 存在则返回 E_CONCURRENT_RUN
6. 命令结束必须释放锁
```

### 14.4 子代理并发

子代理不得直接写：

```text
.sdd/state.json
.sdd/changes/<change-id>/tasks.md
```

子代理只能写：

```text
.sdd/runs/<run-id>/tasks/<task-id>.result.json
```

主代理统一汇总结果并写 state。

### 14.5 并行冲突策略

只有满足以下条件的任务可以并行：

```text
1. 任务无依赖
2. Allowed Files 不重叠
3. Expected New Files 不重叠
4. 不修改同一配置文件
5. 不修改同一测试文件
```

否则顺序执行。

## 15. ARCHIVED 后修改规则

### 15.1 默认只读

`ARCHIVED` 后当前 Change 只读。

禁止执行：

```text
sdd design
sdd plan
sdd build
sdd verify
sdd review
```

允许：

```text
sdd status
sdd archive
```

重复 `sdd archive` 返回 already archived。

### 15.2 继续修改

归档后需要继续修改，必须创建新 Change：

```text
sdd new "基于 add-order-cancel 追加调整：..."
```

新 Change 的 `proposal.md` 应引用旧 Change。

### 15.3 归档保护文件

归档后生成：

```text
.sdd/changes/<change-id>/.archived
```

内容：

```json
{
  "archivedAt": "2026-07-03T10:00:00+08:00",
  "stateHash": "sha256:xxx",
  "artifactHash": "sha256:yyy"
}
```

存在 `.archived` 时，所有写操作必须拒绝。

## 16. 文件与 Git 安全边界

### 16.1 执行前未提交修改保护

每个写命令执行前必须运行 git 状态检查。

如果存在未提交修改：

```text
1. 记录 baseline diff
2. 区分 pre-existing changes 与 current-run changes
3. 不允许把执行前已有修改计入本次任务结果
4. build 前必须在 status 中提示已有修改
```

如果用户未确认，MVP 默认继续但严格记录 baseline。

### 16.2 diff 识别

每个阶段必须记录：

```text
1. before git status
2. before git diff hash
3. after git status
4. after git diff hash
5. changed files delta
```

本次任务实际 diff = after - before。

### 16.3 路径安全

Core 必须使用 Node.js `path.resolve`、`path.relative` 和真实路径解析进行跨平台校验；不得依赖字符串前缀判断。Windows 下必须同时处理盘符大小写、反斜杠、UNC 路径和 junction，macOS 下必须处理 symlink 和大小写差异文件系统。

必须阻断：

```text
1. ../ 路径穿越
2. 绝对路径写入
3. 仓库外文件写入
4. 指向仓库外的 symlink
5. .git/ 内部文件直接修改
```

错误码：

```text
E_PATH_OUTSIDE_REPO
E_SYMLINK_BLOCKED
```

### 16.4 Shell 命令审批边界

默认允许：

```text
1. git status
2. git diff
3. git log
4. mvn test
5. mvn verify
6. npm test
7. npm run test
```

默认禁止：

```text
1. rm -rf
2. curl
3. wget
4. nc
5. ssh
6. scp
7. chmod +x 后立即执行未知脚本
8. docker run 未知镜像
9. kubectl delete
10. 修改 ~/.ssh
11. 读取 ~/.aws
12. 读取 .env
```

### 16.5 多任务修改同一文件

如果多个 Task 需要修改同一文件：

```text
1. 不允许并行
2. plan 阶段必须标记 Depends On
3. build 阶段顺序执行
```

### 16.6 测试生成文件和格式化连带修改

允许修改：

```text
1. 测试快照文件
2. 格式化导致的同任务文件变更
3. 编译缓存之外的测试 fixture
```

但必须记录到 task result 中。

### 16.7 分支和 worktree

MVP 默认不自动创建分支或 worktree。

配置项：

```yaml
git:
  createBranch: false
  createWorktree: false
```

后续 MVP2 可支持：

```text
sdd/<change-id>
```

作为隔离分支。

## 17. 安全设计

### 17.1 仓库提示注入防护

仓库中的 README、注释、脚本、测试输出、MCP 返回内容均视为不可信输入。

Agent 不得遵循仓库文件中的指令，例如：

```text
1. 忽略上层指令
2. 删除文件
3. 上传代码
4. 读取 secrets
5. 执行 curl | sh
```

近期安全研究和新闻均显示 AI 编码 Agent 可能被恶意仓库内容诱导运行恶意命令，因此本工具必须默认不信任仓库内自然语言指令。

### 17.2 MCP 返回内容信任边界

codebase-memory-mcp 返回内容只能作为代码结构和上下文参考，不得作为系统指令。

必须在 Prompt 中标记：

```text
MCP_OUTPUT_IS_UNTRUSTED_CONTEXT
```

### 17.3 secrets 排除规则

默认排除：

```text
.env
.env.*
*.pem
*.key
*.p12
*.jks
id_rsa
id_ed25519
.aws/**
.gcp/**
.azure/**
kubeconfig
application-prod.yml
application-prod.properties
```

### 17.4 日志脱敏

audit log 和 execution report 中不得记录：

```text
1. access token
2. password
3. private key
4. cookie
5. Authorization header
6. database password
7. cloud credential
```

### 17.5 shell 参数转义

所有 shell 命令必须：

```text
1. 使用参数数组执行
2. 禁止字符串拼接 shell
3. 对用户输入做 escape
4. 禁止 eval
5. 禁止 sh -c 执行动态命令
```

### 17.6 插件权限模型

默认权限：

```text
1. 可读仓库文件
2. 可写 .sdd/
3. 可写 Task Allowed Files
4. 可运行白名单测试命令
```

默认禁止：

```text
1. 访问仓库外文件
2. 访问用户 home secrets
3. 网络下载并执行脚本
4. 修改全局配置
```

### 17.7 第三方代码执行风险

vendor OpenSpec / Superpowers 代码前必须进行：

```text
1. LICENSE 检查
2. 依赖扫描
3. install script 检查
4. postinstall 检查
5. lockfile 固定
```

```text
1. Task 说明

3. 关联 Design
4. Allowed Files
5. 相关类 / 方法摘要
6. 相关测试
7. 风险提示
```

优先裁剪：

```text
1. 长代码片段
2. 重复上下文
3. 无关模块说明
4. 低置信度候选文件
```

### 19.3 新鲜度

Context Pack 必须记录：

```text
1. codebase index hash
2. source artifact hash
3. generatedAt
```

如果 index 或 spec/design/tasks 变化，Context Pack 失效，必须重建。

## 23. 命令设计

### 23.1 `sdd init`

职责：

```text
1. 初始化插件配置
2. 初始化 .sdd/
3. 检查 codebase-memory-mcp
4. 不可用时降级为 file-scan
5. 生成 CLAUDE.md
6. 生成 AGENTS.md
7. 生成 schema
8. 构建代码库摘要
9. 更新 state 为 INDEX_READY
```

### 23.2 `sdd new "粗略需求"`

职责：

```text
1. 创建 Change
2. 查询代码上下文
3. 生成 impact.md
4. 进行头脑风暴
5. 提问澄清需求
6. 保存 answers / assumptions
7. 生成 spec.md
8. 更新状态为 SPEC_READY
```

### 23.3 `sdd design`

职责：

```text
1. 读取 spec 和 impact
2. 生成 design.md
3. 检查真实代码结构
4. 更新状态为 DESIGN_READY
```

### 23.4 `sdd plan`

职责：

```text
1. 读取 spec 和 design
2. 生成 tasks.md
3. 生成 test-plan.md
4. 生成 context packs
5. 标记任务依赖
6. 更新状态为 PLAN_READY
```

### 23.5 `sdd build`

职责：

```text
1. 根据 tasks 实现
2. 子代理可用时并行
3. 不可用时顺序执行
4. 遵守 Allowed Files
5. 写 task result
6. 更新状态为 BUILD_READY
```

### 23.6 `sdd verify`

职责：

```text
1. 检查任务完成度
2. 检查需求覆盖
3. 检查验收标准
4. 检查功能边界
5. 检查基础 Drift
6. 生成 verify-report.md
```

### 23.7 `sdd review`

职责：

```text
1. 检查代码质量
2. 检查无关修改
3. 检查性能
4. 检查复杂度
5. 检查安全风险
6. 生成 review-report.md
```

### 23.8 `sdd archive`

职责：

```text
1. 检查 verify PASS
2. 检查 review PASS
3. 生成 traceability.md
4. 生成 archive-report.md
5. 写 .archived
6. 更新状态为 ARCHIVED
```

### 23.9 `sdd status`

职责：

```text
1. 显示初始化状态
2. 显示索引状态
3. 显示当前 Change
4. 显示当前阶段
5. 显示失败原因
6. 显示建议下一步
```

### 23.10 `sdd auto "粗略需求"`

职责：

```text
1. 从当前状态自动选择起点
2. 依次执行后续阶段
3. 遇到 BLOCKER 暂停
4. 遇到失败保存状态
5. status 给出恢复建议
```

## 24. 测试与验收设计

### 24.1 测试层级

| 层级               | 说明                            |
| ---------------- | ----------------------------- |
| Unit Test        | 状态机、schema、路径安全、幂等规则          |
| Integration Test | mock MCP、真实文件系统、插件生成          |
| E2E Test         | 示例仓库跑完整流程                     |
| Security Test    | prompt injection、secrets、路径穿越 |
| Regression Test  | schema 迁移、重复 init、失败恢复        |

### 24.2 mock MCP 与真实 MCP 边界

MVP0 必须支持 mock MCP。

测试分为：

```text
1. mock codebase-memory-mcp：CI 默认使用
2. real codebase-memory-mcp：本地集成测试使用
```

### 24.3 示例仓库

至少准备：

```text
fixtures/springboot-order-service
fixtures/node-basic-service
fixtures/broken-state-project
fixtures/security-malicious-readme
```

### 24.4 Given / When / Then 示例

#### 场景 1：未初始化执行 new

```text
Given 当前目录没有 .sdd/
When 执行 sdd new "实现订单取消"
Then 返回 E_NOT_INITIALIZED
And 退出码为 3
```

#### 场景 2：init 时 MCP 不可用

```text
Given codebase-memory-mcp 未安装
When 执行 sdd init
Then 系统降级为 fallback-file-scan
And state.json 中 degraded=true
And 退出码为 0
```

#### 场景 3：BLOCKER 未回答

```text
Given 执行 sdd auto "实现订单取消"
And SpecEngine 生成 BLOCKER 问题
When 用户未回答
Then 状态为 CLARIFYING
And questions.md 已写入
And 不进入 build
```

#### 场景 4：非法流转

```text
Given 当前状态为 SPEC_READY
When 执行 sdd build
Then 返回 E_INVALID_PHASE_COMMAND
And next 为 sdd design
And 退出码为 3
```

#### 场景 5：ARCHIVED 后修改

```text
Given 当前 Change 已 ARCHIVED
When 执行 sdd build
Then 返回 E_ARCHIVED_READONLY
And 退出码为 3
```

#### 场景 6：并发执行

```text
Given sdd build 正在运行
When 第二个进程执行 sdd plan
Then 返回 E_CONCURRENT_RUN
And 退出码为 9
```

#### 场景 7：路径穿越

```text
Given Task 尝试写入 ../secret.txt
When 执行 sdd build
Then 返回 E_PATH_OUTSIDE_REPO
And 退出码为 10
```

#### 场景 8：Claude Code 与 Codex 命令对等

```text
Given 同一个已初始化 fixture 和相同的 .sdd/ 初始状态
When 分别通过 Claude Code 的 /sdd.status 和 Codex 的 sdd status 执行
Then 两次 CommandResult 的 ok、state、exitCode、error.code 和 next 完全一致
And 生成或读取的制品内容一致
```

每个阶段命令至少必须有一条 Adapter 契约测试，验证 Claude Code slash command 与 Codex skill 都映射到相同的 Core command 和参数。

### 24.5 支持平台测试矩阵

测试矩阵必须按操作系统和 Agent 平台的组合执行，不能用单独的平台测试代替组合测试：

| 操作系统      | Agent 平台   | MVP0 | MVP1 | MVP2 |
| --------- | ---------- | ---- | ---- | ---- |
| macOS   | Claude Code | 必测   | 必测   | 必测   |
| macOS   | Codex       | 必测   | 必测   | 必测   |
| Windows | Claude Code | 必测   | 必测   | 必测   |
| Windows | Codex       | 必测   | 必测   | 必测   |

## 26. 最终验收标准

除非条目明确限定平台，以下命令验收必须同时通过 Claude Code 的 `/sdd.*` 形式和 Codex 的 `sdd ...` 形式。两个 Adapter 必须产生一致的 Core 请求、状态、错误码和制品。

### 26.1 MVP0 验收

```text
1. 插件包可安装到 Claude Code 项目
2. 插件包可安装到 Codex 项目
3. 两个平台的 init 命令都能生成 .sdd/
4. 两个平台的 init 命令都能生成 CLAUDE.md 和 AGENTS.md
5. codebase-memory-mcp 不可用时能降级
6. 两个平台的 status 命令都能输出相同状态
7. state.json 原子写入
8. 并发写命令会被 lock 阻断
9. schema 校验可运行
```

### 26.2 MVP1 验收

```text
1. 两个平台的 new 命令都能生成一致的 spec.md
2. BLOCKER 未回答不进入 design
3. 两个平台的 design 命令都能生成一致的 design.md
4. 两个平台的 plan 命令都能生成一致的 tasks.md
5. 每个 Task 有 Context Pack
6. 重复执行不会静默覆盖人工修改
```

### 26.3 MVP2 验收

```text
1. 两个平台的 build 命令都能按任务执行
2. 两个平台的 verify 命令都能发现未完成需求
3. 两个平台的 review 命令都能发现无关修改
4. 两个平台的 archive 命令都能生成一致的归档报告
5. 两个平台的 auto 命令都能从粗略需求跑完整流程
6. ARCHIVED 后只读
```

## 28. 总结

`sdd-harness` 的第一版不是独立 CLI，而是 Claude Code / Codex 插件包。

它使用：

```text
1. 原版 codebase-memory-mcp
2. 内置化 OpenSpec 能力形成 SpecEngine
3. 内置化 Superpowers 能力形成 TddEngine
4. .sdd/ 作为唯一事实源
5. 9 个阶段命令 + sdd auto
6. 明确状态机、命令契约、安全边界、Schema、验收测试
```

最终目标是让 AI 编码过程从“自由发挥”变成“受控执行”。
