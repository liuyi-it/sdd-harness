## 2. 总体验收结论

**结论：部分通过，不建议标记为 4 期完全验收通过。**

当前版本已经完成 4 期的大部分结构性目标：保留 `sdd auto` 作为唯一公开 Loop 入口，不新增 `sdd loop` 命令；新增 `BUILD_WAITING_AGENT`；引入 `LoopEngine`、`DecisionEngine`、`LoopEventStore`；并增强 `build next / build complete` 的 handoff 与安全校验能力。4 期验收摘要本身要求覆盖 AC-001 到 AC-034，包括唯一入口、handoff、build 安全校验、loop event、CLI flags、schema migration、v2 result 等内容。

但当前版本仍存在影响验收的关键缺口，尤其是 `auto --resume` 在 `BUILD_WAITING_AGENT` 状态下不能保证返回同一 handoff，`auto --events / --loop-status` 的文本输出不足，`runId` 缺少路径净化，以及 `auto --resume/restart/stop` 未按设计持有 FileLock。

## 4. 已满足的核心能力

### 4.1 `sdd auto` 保持唯一公开 Loop 入口

CLI 命令列表包含 `auto`，未新增公开 `loop` 命令；`loop` 只是 `status --loop` 的选项。CLI help 中也扩展了 `auto --resume / --restart / --stop / --events`。

### 4.2 `Core.runAuto()` 已完成薄封装

当前 `Core.runAuto()` 已经只负责构造 `LoopEngine` 并调用 `loopEngine.run(request)`，符合 4 期架构目标。

### 4.3 `BUILD_WAITING_AGENT` 状态已纳入契约

`PHASES` 已新增 `BUILD_WAITING_AGENT`，位于 `BUILDING` 与 `BUILD_READY` 之间。

### 4.4 `build next` handoff 已基本正确

`buildNextTask()` 在新分配任务时会把 `currentPhase` 写为 `BUILD_WAITING_AGENT`，写入 `activeLoop.status = WAITING_AGENT` 和 `activeLoop.waiting`，并返回 `AgentActionRequired`。

重复执行 `build next` 时，如果当前已经在 `BUILD_WAITING_AGENT` 且存在 `activeLoop.waiting.taskId/resultFile`，会返回同一份 handoff，不重复分配任务。

### 4.5 `build complete` 安全校验已有明显增强

当前 `build complete` 已覆盖：

- `result.taskId` 与 `--task` 一致性校验。
- 当前等待任务校验。
- `validateTaskFiles()` 文件范围校验。
- TDD evidence 命令安全校验。
- verification 命令安全校验。
- verification failed 返回 `E_VERIFY_FAILED`。

相关逻辑已在 `buildCompleteTask()` 中实现。

### 4.6 `PLAN_READY` next 已切换为 `sdd build next`

`status.ts` 中 `PLAN_READY` 对应的 next 已经是 `sdd build next`。

### 4.7 unchanged 状态收敛已修复

`design` unchanged 分支会写回 `DESIGN_READY`、清理 `inProgressPhase`、设置 `suggestedCommand = sdd plan`。

`plan` unchanged 分支会写回 `PLAN_READY`、清理 `inProgressPhase`、设置 `suggestedCommand = sdd build next`。

## 5. 未通过或条件通过项

### 5.1 AC-009：`auto --resume` 等待 Agent 时返回同一 handoff

当前 `LoopEngine.run()` 会把 `resume` 分发到 `resumeAuto()`，然后进入 `runAuto()`。但 `runAuto()` 对 `BUILD_WAITING_AGENT` 没有专门处理，`autoCommand()` 也没有映射该状态，因此可能直接 finalize 并返回 status，而不是调用 `build next` 返回同一 handoff。

这是当前最影响 4 期闭环验收的问题。

**验收建议：必须补测试。**

测试应覆盖：

```bash
sdd auto
# 返回 BUILD_WAITING_AGENT + actionRequired
sdd auto --resume
# 必须返回相同 taskId/contextPack/resultFile
```

### 5.2 AC-024 到 AC-027：loop events / loop status 文本可观测性不足

`getEvents()` 和 `getLoopStatus()` 已经把内容放入 `data`。 但 CLI 文本输出只显示 `State / Change / Next / Warning / Action Required / Error`，没有 events 或 activeLoop 专用渲染。

因此 JSON 输出可用，但普通文本输出不足。对使用 CLI 的 Agent 或开发者来说，`sdd auto --events` 可能只看到状态，看不到事件列表。

**验收建议：条件不通过，需补文本渲染和 golden test。**

### 5.3 并发安全未完全满足设计要求

设计文档明确要求 `build next/complete`、`auto --resume/restart/stop` 持有 FileLock。 当前 `build next` 和 `build complete` 有 FileLock，但 `LoopEngine.resumeAuto()`、`restartAuto()`、`stopAuto()` 没有看到显式锁保护。

**验收建议：并发验收不通过，需补 FileLock。**

### 5.4 `runId` 缺少路径净化

`LoopStore.readRun()`、`writeRun()`、`hasRun()` 直接用 `runId` 拼接文件路径。 `LoopEventStore` 也直接使用 `runId` 读写 JSONL。

`--run` 参数来自 CLI，最终可进入 `resumeAuto()` 读取 run 文件。

**验收建议：安全验收不通过，需增加 `runId` 白名单校验。**

### 5.5 `build complete` result schema 仍需强化

当前 `build complete` 只做浅层字段校验：`schemaVersion`、`taskId`、`status`。 后续将 `modifiedFiles` 直接当作 `string[]` 使用。 如果 result 文件中 `modifiedFiles` 为 `[123]` 或其他畸形结构，可能触发普通 TypeError，而不是稳定返回 `CommandResult`。

**验收建议：条件通过，但发布前建议用 Zod 补完整 schema。**

## 6. 最小补齐清单

要把当前版本推进到“4 期验收通过”，建议至少补以下修改：

1. `LoopEngine.runAuto()` 在 `BUILD_WAITING_AGENT` 状态下调用 `build next` 并返回同一 `actionRequired`。
2. `PAUSE_FOR_AGENT` finalize 时保持 `WAITING_AGENT`，不要覆盖成 `PAUSED` 或 `RUNNING`。
3. `LoopStore` / `LoopEventStore` / `resumeAuto()` 对 `runId` 做白名单校验。
4. `auto --resume/restart/stop` 增加 FileLock。
5. `outputText()` 增加 events / activeLoop 渲染。
6. `build complete` 增加完整 result schema 校验。
7. 增加以下回归测试：
   - `auto --resume` from `BUILD_WAITING_AGENT` returns same handoff。
   - `sdd auto` from `BUILD_WAITING_AGENT` 不改变 `activeLoop.status`。
   - `sdd auto --events` 文本输出包含事件。
   - `sdd auto --loop-status` 文本输出包含 runId/status/waiting。
   - `--run ../../xxx` 返回 `E_SECURITY_BLOCKED`。
   - 畸形 `modifiedFiles: [123]` 稳定返回错误，不抛普通异常。
