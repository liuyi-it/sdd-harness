## 1. 已修复项

### 1.1 `BUILD_WAITING_AGENT` 已纳入 auto 阶段映射

`COMMAND_BY_PHASE` 已包含：

```ts
BUILD_WAITING_AGENT: "build";
```

并且 `runAuto()` 对 `build` 会注入 `subcommand: "next"`，所以从 `BUILD_WAITING_AGENT` 再次进入时，理论上会调用 `build next` 返回同一 handoff。

`PAUSE_FOR_AGENT` 现在会将最终状态设为 `WAITING_AGENT`，`finalizeLoop()` 也允许 `WAITING_AGENT`。

`resumeAuto()`、`restartAuto()`、`stopAuto()` 都已经创建并申请 FileLock。

## 2. 新发现的 P0 问题

### P0-1：`sdd auto` 外层 FileLock 与内部命令 FileLock 嵌套，auto 可能无法推进

最新版在 `LoopEngine.run()` 的普通 auto 路径上加了全局 FileLock：

```ts
const lock = new FileLock(this.root);
await lock.acquire("sdd auto");
try {
  return this.runAuto(request);
} finally {
  await lock.release();
}
```

但 `runAuto()` 内部会调用 `this.execute()` 执行实际命令，例如 `build`。 `Core.execute()` 再进入 `runBuild()`。

问题是 `build next` 本身也会申请 FileLock：

```ts
const lock = new FileLock(root);
await lock.acquire("sdd build next", ...)
```

`new` 命令也会申请 FileLock。 所以 `sdd auto <需求>` 从 `INDEX_READY` 开始时，第一步执行 `new` 就可能撞上自己持有的 `sdd auto` 锁；从 `PLAN_READY` 开始时，执行 `build next` 也会撞锁。

当前 `FileLock` 不支持同进程重入。它看到已有锁后，会判断旧锁未过期或旧 pid 仍存活，然后抛 `E_CONCURRENT_RUN`。

**影响：这是阻断级问题。最新版很可能导致正常 `sdd auto` 无法跑通。**

修复建议：

不要在 `LoopEngine.run()` 外层长时间持有 FileLock 包裹整个 `runAuto()`。可选方案：

```ts
// 不要这样：
await lock.acquire("sdd auto");
return this.runAuto(request);

// 改为：
return this.runAuto(request);
```

`runAuto()` 内部调用的 `new/design/plan/build/verify/review/archive` 已经各自负责写锁。LoopEngine 自己写 loop run/event/state 的地方，可以单独短锁，不能覆盖整个命令执行链。

`resumeAuto()` / `restartAuto()` 也不要在持锁状态下直接 `return this.runAuto(...)`，否则同样会嵌套锁。应当：

1. 申请锁。
2. 修改 activeLoop / run 状态。
3. 释放锁。
4. 再进入 `runAuto()`。

## 3. 仍未完全解决的问题

### P1-1：`auto --resume <run-id>` 仍可能丢失 waiting handoff

`resumeAuto()` 在传入具体 `runId` 时，会直接把 `activeLoop` 覆盖为：

```ts
{
  loopId: run.loopId,
  runId: run.runId,
  status: "RUNNING"
}
```

没有保留 `waiting`。

而 `build next` 的“返回同一 handoff”依赖 `activeLoop.waiting.taskId/resultFile` 存在。

如果用户在 `BUILD_WAITING_AGENT` 状态下执行：

```bash
sdd auto --resume --run run-xxx
```

可能出现：

1. `activeLoop.waiting` 被清空。
2. `build next` 找不到当前 waiting。
3. 当前任务状态已经是 `BUILDING`，会被排除。
4. 最终返回 `BUILDING / pendingBuild`，而不是同一份 handoff。

修复建议：

- 如果 `resumeRunId === current.activeLoop.runId`，不要覆盖 activeLoop，直接保留原对象。
- 如果 resume 到历史 run，应从 run 的最后一个 `AGENT_HANDOFF` step 中恢复 `waiting`，或明确拒绝恢复缺失 waiting 信息的 run。

### P1-2：`build complete` result schema 仍偏弱

当前 schema 校验仍只是浅校验：

```ts
schemaVersion;
taskId;
status in VALID_TASK_STATUSES;
```

后续代码直接把 `modifiedFiles` 当 `string[]`，把 `tddEvidence` 当数组，verification 也当数组使用。

如果 result 文件中出现：

```json
{
  "schemaVersion": "1.2.0",
  "taskId": "T-001",
  "status": "SUCCEEDED",
  "modifiedFiles": [123],
  "tddEvidence": {},
  "verification": {}
}
```

仍可能触发普通运行时异常，而不是稳定返回 `CommandResult` 错误。

修复建议：用 Zod 或手写 guard 校验完整结构：

- `modifiedFiles?: string[]`
- `createdFiles?: string[]`
- `commandsRun?: string[]`
- `tddEvidence?: Array<{ phase: string; passed: boolean; command?: string }>`
- `verification?: Array<{ command: string; passed: boolean }>`

### P2-1：`LoopEventStore.read()` 仍吞掉所有错误

事件读取失败时，无论是文件不存在、JSONL 损坏，还是权限错误，都会直接返回 `[]`。

建议只对 `ENOENT` 返回空数组；JSON parse 失败应返回明确错误，至少写 warning，否则事件日志损坏不可观测。

### P2-2：CLI help 仍未展示 `auto --loop-status`

CLI 已支持 `--loop-status` 参数解析，但 help 文本只展示了 `auto --events`，没有展示 `auto --loop-status`。

这是文档/可发现性问题，不阻断核心功能。

### P2-3：`auto --stop` 返回 `ok: true` 但 `state: "FAILED"`

`stopAuto()` 成功停止后返回：

```ts
ok: true,
state: "FAILED",
exitCode: 0
```

这不一定破坏功能，但语义不清。建议返回当前真实 `currentPhase`，或引入更明确的 `data.loopStatus = "ABORTED"`。
