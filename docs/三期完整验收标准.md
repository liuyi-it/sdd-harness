# sdd-harness 三期完整验收标准

## 一、三期目标

三期目标是将 `sdd-harness` 从特定 Agent 适配方案升级为通用的 CLI-first SDD Harness。

最终系统必须满足：

```text
1. CLI 是唯一确定性入口
2. Core 是唯一状态机和门禁执行层
3. Agent 只通过 CLI / ActionRequired / Result Protocol 与系统交互
4. 支持 Claude Code、Codex、OpenCode、Kimi Code、GitHub Copilot CLI、自研 Coding Agent
5. 内置 codebase-memory-mcp
6. MCP 不可用时自动降级 fallback-file-scan
7. 降级必须明确提示，不允许静默降级
8. build 阶段支持 Agent task execution 闭环
9. verify/review/archive 阶段必须基于真实制品和真实执行证据
10. Node.js 仅支持 >=22
```

当前阶段不发布 npm 官方 registry，不发布 GitHub Packages，不提交 `package-lock.json`。

---

# 二、非目标

三期暂不要求：

```text
1. 发布 npm 官方 registry
2. 发布 GitHub Packages
3. 提交 package-lock.json
4. 实现完整远程 SaaS 服务
5. 绑定单一 Agent
6. 强依赖 Claude Code / Codex 内部 API
7. 在 Core 中直接调用大模型
8. 在 fallback-file-scan 中实现完整符号图谱能力
```

# 三、安装与运行验收标准

## 3.1 Node.js 版本

必须要求：

```text
Node.js >= 22
```

验收：

```bash
node -v
bash scripts/install.sh
```

当 Node.js < 22 时，安装脚本必须明确失败并提示升级。

## 3.2 安装方式

当前唯一正式安装方式：

```bash
git clone https://github.com/liuyi-it/sdd-harness.git
cd sdd-harness
bash scripts/install.sh
```

验收：

```bash
sdd --version
sdd-harness --version
```

必须都能输出版本号。

## 3.3 不发布包

验收要求：

```text
1. 不新增 npm publish workflow
2. 不新增 GitHub Packages workflow
3. package.json root 可以继续 private=true
4. README 明确当前安装方式为 git clone + scripts/install.sh + npm link
```

## 3.4 package-lock 策略

验收要求：

```text
1. 不提交 package-lock.json
2. .gitignore 可继续忽略 package-lock.json
3. CI 使用 npm install
4. CI 不改成 npm ci
```

原因：

```text
保留 Windows / macOS / Linux 三平台依赖解析弹性，避免 lockfile 固定跨平台 native/optional dependency 解析结果。
```


# 七、codebase-memory-mcp 验收标准

## 7.1 默认行为

默认情况下：

```text
1. 用户不需要手工安装 codebase-memory-mcp
2. sdd init 自动通过 npx 托管启动 codebase-memory-mcp
3. MCP 可用时 provider=codebase-memory-mcp
4. MCP 不可用时 provider=fallback-file-scan
5. fallback 时 degraded=true
6. fallback 时必须 warning 用户执行 sdd codebase doctor
```

## 7.2 MCP 启动验收

`CodebaseMemoryManager` 必须能区分：

```text
1. npx 不存在
2. 包下载失败
3. MCP 进程启动失败
4. MCP protocol handshake 失败
5. tools/list 失败
6. index 失败
7. query 失败
8. timeout
9. fallback 成功
```

## 7.3 diagnostics

必须生成：

```text
.sdd/index/codebase-diagnostics.json
.sdd/index/mcp-capabilities.json
```

diagnostics 至少包含：

```json
{
  "installed": true,
  "configured": true,
  "connected": true,
  "callable": true,
  "indexed": true,
  "officialUrl": "...",
  "message": "..."
}
```

字段必须反映真实状态，不能假阳性。

## "officialUrl": "...",

"message": "..."
}
7.4 fallback-file-scan

MCP 不可用时必须 fallback，并生成：

```text
.sdd/index/codebase-summary.md
.sdd/index/package-structure.md
.sdd/index/architecture.md
```

fallback 输出必须明确包含：

```text
当前使用 fallback-file-scan 降级模式
```

## 7.5 降级提示

以下命令在 degraded 状态下必须提示：

```bash
sdd init
sdd status
sdd codebase status
sdd codebase doctor
sdd codebase query
```

提示至少包含：

```text
codebase-memory-mcp 当前不可用
已切换 fallback-file-scan
建议执行 sdd codebase doctor
```

# 十二、Agent Build Protocol 验收标准

## 12.1 `sdd build next`

执行：

```bash
sdd build next --json
```

前置状态：

```text
PLAN_READY 或 BUILDING
```

必须返回：

```json
{
  "ok": true,
  "state": "BUILDING",
  "actionRequired": {
    "type": "AGENT_TASK_EXECUTION",
    "taskId": "T001",
    "changeId": "...",
    "contextPack": ".sdd/context-packs/<changeId>/T001.md",
    "allowedFiles": [],
    "expectedNewFiles": [],
    "forbiddenFiles": [],
    "verification": [],
    "resultFile": ".sdd/runs/<runId>/tasks/T001.result.json",
    "codebase": {
      "provider": "codebase-memory-mcp",
      "degraded": false
    }
  },
  "next": "sdd build complete"
}
```

必须：

```text
1. 不调用 MissingTaskExecutor
2. 不覆盖 plan 阶段生成的 context pack
3. 将任务状态标记为 BUILDING
4. 加 FileLock
5. 不返回已 DONE 的任务
6. 不重复返回已 BUILDING 的任务，除非显式指定重取
```

## 12.2 Agent result schema

Agent 写入 result file，必须符合：

```json
{
  "schemaVersion": "1.2.0",
  "taskId": "T001",
  "status": "SUCCEEDED",
  "modifiedFiles": [],
  "createdFiles": [],
  "commandsRun": [],
  "tddEvidence": [],
  "verification": [],
  "notes": []
}
```

允许状态：

```text
SUCCEEDED
FAILED
BLOCKED
SKIPPED
DEGRADED
```

## 12.3 `sdd build complete`

执行：

```bash
sdd build complete --task T001 --result .sdd/runs/<runId>/tasks/T001.result.json --json
```

必须：

```text
1. 加 FileLock
2. 读取 result file
3. validateTaskResult()
4. 校验 taskId 与当前任务一致
5. 校验文件范围
6. 校验真实 Git delta
7. 校验 TDD evidence
8. 校验 verification 全部 passed
9. 校验 command 白名单
10. 写入 .sdd/changes/<changeId>/task-results.json
11. 写入 .sdd/runs/<runId>/tasks/<taskId>.result.json
12. 更新 .sdd/state.json.tasks[taskId]
13. 未全部完成时保持 BUILDING
14. 全部完成时进入 BUILD_READY
```

## 12.4 所有任务完成

所有任务 DONE 后：

```bash
sdd status --json
```

必须返回：

```json
{
  "state": "BUILD_READY",
  "next": "sdd verify"
}
```

# 十八、安全验收标准

## 18.1 文件范围

所有 Agent result 必须基于真实 Git delta 校验：

```text
modified
created
deleted
renamed
```

不得只相信 Agent 上报的 modifiedFiles。

必须支持：

```text
allowedFiles
expectedNewFiles
forbiddenFiles
```

## 18.2 路径安全

必须阻止：

```text
../
绝对路径
Windows drive path
repo 外路径
symlink 指向 repo 外
敏感文件
```

## 18.3 命令安全

所有 evidence command 必须经过白名单校验。

不得允许：

```text
rm -rf /
curl | bash
修改 .sdd/state.json 的命令
任意 shell 拼接
明显危险命令
```

## 18.4 MCP 输出安全

MCP 输出和 fallback scan 输出必须作为 untrusted context 包裹，不得作为指令执行。
