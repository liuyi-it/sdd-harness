# Agent-native 交互体验规格与长期目标

> 状态：长期演进规格
>
> 本文用于约束后续工作流、Adapter 与 CLI 的优化方向。任何提升自动化或减少界面噪声的改动，都不得削弱 Core 的状态机、质量门禁、安全边界与可追溯性。

## 1. 初心与北极星

sdd-harness 的初心不是把开发过程变成一串命令、JSON 和权限申请，而是让 Agent 在可验证、可审计、可恢复的约束下，稳定地帮助用户完成真实的软件变更。

北极星体验是：

> 用户像与一位可靠的工程师协作一样提出需求、回答必要问题、确认重要影响并获得结果；流程治理在后台完成，默认不打扰用户。

这意味着：

- **对用户自然**：用业务语言交流，先给结论，只在需要决策时提问。
- **对 Agent 顺滑**：上下文、任务约束和阶段转移由 Adapter 内部衔接，不要求 Agent 手工搬运中间协议。
- **对系统严格**：所有变更仍经由 Core，保留文件范围、验证、审查、锁、状态恢复和归档证据。

## 2. 产品目标

### 2.1 默认协作模式

默认模式采用 Agent-native 体验。用户只应看到以下四类状态：

| 用户可见状态 | 用户应看到的内容                               | 例子                                 |
| ------------ | ---------------------------------------------- | ------------------------------------ |
| 已理解并推进 | 目标、预计影响、当前工作                       | “我会在订单服务补充取消校验和测试。” |
| 需要信息     | 一个或少量业务问题，以及可直接回答的示例       | “到期日按合同还是产品计算？”         |
| 需要确认     | 会扩大范围、产生外部副作用或存在明显风险的操作 | “需要新增依赖，是否继续？”           |
| 已完成或受阻 | 完成内容、验证结果；或可理解的阻塞原因与下一步 | “测试通过；还需要你确认删除策略。”   |

默认不展示以下内部实现细节：

- CLI JSON、`CommandResult`、阶段码、任务/运行/loop ID；
- `.sdd/` 路径、Context Pack、Policy Bundle、TaskExecutionResult；
- 仅供诊断使用的错误码、调试字段和底层工具输出。

用户明确要求原始输出、审计证据或排障信息时，才按请求展示相应细节。

### 2.2 严格治理仍是默认能力

体验简化不等于跳过治理。每一次实际变更仍必须：

1. 经由 Core 校验阶段、锁和活动 Change；
2. 受任务文件范围与安全规则约束；
3. 执行与记录必要验证；
4. 在 `verify`、`review`、`archive` 中保留可追溯证据；
5. 在失败或中断后可恢复，不伪造成功。

## 3. 目标架构

当前模型要求 Agent 多次运行 CLI、读取 Context Pack、写入 TaskExecutionResult 并提交 `build complete`。这对审计友好，但把编排协议暴露给了宿主界面，造成重复工具卡片和权限申请。

目标模型如下：

```text
用户
  ↕ 自然语言：目标、问题、确认、结果
Adapter / Agent-native Executor
  ↕ 内部编排：上下文注入、任务结果归一化、阶段推进
Core
  ↕ 事实与治理：状态、制品、范围、验证、审查、归档
```

职责边界：

- **Adapter / Agent-native Executor**：将 Core 结果转换为用户语言；在内部读取 Context Pack、承接任务证据并调用下一阶段；合并可合并的本地操作。
- **Core**：仍是唯一事实源，不能被 Adapter 绕过或直接修改 `.sdd/state.json`。
- **CLI**：保留为 CI、脚本、调试和严格审计接口；不再是日常对话中每一步都必须显式操作的主交互面。

## 4. 权限与工具调用原则

宿主应用决定哪些 shell、文件和网络操作需要授权；sdd-harness 不能承诺隐藏或绕过这些安全提示。但应通过编排减少无意义的提示次数。

### 4.1 应尽量在内部完成

- 多阶段状态读取与确定性阶段推进；
- Context Pack 的生成、读取和传递；
- TaskExecutionResult 的结构化归一化和落盘；
- 非破坏性的本地检查、摘要与证据收集。

### 4.2 必须显式浮出给用户

- 首次实际修改业务文件，或宿主要求的写入授权；
- 新增依赖、网络访问、外部服务调用、推送/创建 PR；
- 删除、覆盖、迁移或明显扩大文件范围的操作；
- 无法从代码或既有约定推断的业务决策。

不得为了减少提示而合并不相关权限、隐瞒副作用或放宽文件/命令白名单。

## 5. 命令与协议的定位

`sdd new`、`sdd auto`、`build next`、`build complete`、`--answers` 等是**内部工作流协议**，不是默认的用户对话语言。

Adapter 的要求：

- 只能将必要的自然语言问题交给用户；
- 不要求用户理解 Change、Phase、Context Pack 或答案 JSON；
- 不把 CLI 的 `--non-interactive` 作为默认参数；
- 仅在用户明确选择 CLI、CI、原始输出或诊断模式时暴露精确命令。

## 6. 模式划分

| 模式         | 使用场景                     | 对用户暴露内容                       |
| ------------ | ---------------------------- | ------------------------------------ |
| 默认协作模式 | 日常需求、开发与修复         | 四类用户状态、必要确认和最终验证摘要 |
| 严格审计模式 | CI、合规、复杂迁移、排障     | 命令、完整证据、状态与制品位置       |
| 诊断模式     | 工作流异常、Adapter/MCP 排障 | 原始输出、错误码、运行记录与诊断建议 |

模式切换必须由用户显式请求、项目配置或 CI 环境决定，不能在普通对话中静默降级为“少审计”。

## 7. 验收标准与度量

### 7.1 交互验收

- 普通完整需求从开始到首次代码修改前，用户不看到 JSON、任务 ID、Context Pack 或阶段码。
- 需求不清时，单轮只提出少量可回答的业务问题，不解释内部门禁实现。
- 完成时只汇报变更、验证结果、剩余风险与下一步；原始证据按需提供。
- 控制命令、子命令和内部参数不能作为普通对话的必填知识。

### 7.2 治理验收

- 所有业务文件修改仍可追溯到 Change、任务、验证与审查证据。
- Adapter 内部编排失败后，Core 状态可恢复且不出现伪成功。
- 严格审计模式可以重建默认模式省略的执行细节。
- 减少工具调用不得扩大允许文件、允许命令或网络权限。

### 7.3 可观测指标

- 每个普通 Change 的用户可见工具/权限提示次数；
- 每个普通 Change 的用户可见内部字段数量（目标为 0）；
- 从需求提出到首次有效代码操作的中断次数；
- 因协议参数错误（空需求、错误子命令、误用 `--non-interactive`）导致的失败率；
- 完成后 verify/review/归档证据完整率。

## 8. 演进顺序

1. **收口交互协议**：所有 Adapter 统一用户可见状态与内部字段屏蔽规则。
2. **Adapter 内部执行器**：将 Context Pack 读取、任务结果归一化和阶段续跑从对话层收回 Adapter。
3. **合并确定性操作**：减少不必要的 CLI 往返与独立工具调用，同时保留 Core 校验点。
4. **引入模式开关与指标**：默认协作模式、严格审计模式与诊断模式可验证切换。
5. **以真实任务回归**：用需求澄清、普通功能、失败恢复、依赖变更和迁移任务验证体验与治理均未退化。

## 9. 非目标

- 不绕过宿主应用的安全授权机制；
- 不把 Core 状态机改成可由 Prompt 自由决定的流程；
- 不删除 `.sdd/` 中的审计证据；
- 不以“界面更干净”为理由自动执行高风险或外部副作用操作；
- 不要求用户学习 CLI 才能完成普通开发协作。

## 10. 每次优化前的检查问题

实施任何相关改动前，必须回答：

1. 这项信息是用户决策所必需，还是仅供系统编排？
2. 这次工具调用是否真的需要独立授权，能否安全地在内部合并？
3. 简化后，Core 的状态、范围、验证和恢复证据是否仍然完整？
4. 用户是否仍能在需要时获得完整审计与诊断信息？
5. 这项改动是否让 Agent 更像可靠工程师，而不是要求用户操作工作流引擎？

若任一答案不清楚，优先保持治理能力，并先补足设计与验证，而不是以隐藏信息替代真正的体验改进。
