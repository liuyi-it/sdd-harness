# sdd-harness 工程能力策略层接入方案

## 1. 文档定位

本方案用于在不新增公开命令、不改变现有状态机、不破坏 `auto` 与手动模式一致性的前提下，将以下上游项目中的工程实践融入 `sdd-harness`：

- `obra/superpowers`
- `mattpocock/skills`

接入后的目标不是让用户调用更多 Skill，而是让现有命令在每个阶段自动使用更稳定的工程方法。

---

## 2. 背景与结论

### 2.1 当前 sdd-harness 的边界

`sdd-harness` 当前已经明确划分：

- CLI/Core 负责状态机、Schema、Git delta、安全范围和质量门禁；
- Agent 负责需求分析、方案设计、任务拆解、编码和测试；
- `.sdd/` 是工作流唯一事实源；
- `auto` 与手动命令最终都进入相同的 Core 阶段实现。

当前 Adapter 会向 Claude Code、Codex、OpenCode 写入 Skill、命令模板和仓库指令，但相同 operating rules 在多个位置重复。

### 2.2 两个上游项目的定位

Superpowers 是完整的软件开发方法论，包含需求讨论、设计确认、计划编写、worktree、TDD、子 Agent 执行、代码审查和分支收尾。

mattpocock/skills 更偏向小型、可组合、易改造的工程 Skill，并强调渐进披露、单一事实源和可检查完成条件。

### 2.3 最终决策

采用三层结构：

```text
第一层：sdd-harness Core / LoopEngine
负责唯一流程编排、状态流转和门禁

第二层：Superpowers 已适配协议
负责 Task、TDD、Context Pack、Verification discipline

第三层：精选 mattpocock/skills 方法
负责设计质量、任务切片、双轴审查、Prompt 结构和上下文管理
```

不安装或运行完整上游插件。

不允许上游 Skill：

- 决定下一阶段；
- 直接修改 `.sdd/state.json`；
- 自行创建 worktree；
- 自行提交代码；
- 创建第二套任务状态；
- 创建外部 Issue/Ticket；
- 要求新增公开命令；
- 绕过 Core 的验证结果。

---

# 3. 核心目标

## 3.1 功能目标

1. 保留当前全部公开命令，命令数量不增加。
2. `auto` 模式在需求澄清结束后继续无人值守执行。
3. 手动模式允许用户按阶段执行现有命令。
4. auto 和手动模式使用完全相同的阶段 Policy。
5. 增强设计、计划、构建、验证、审查和失败恢复质量。
6. 所有 Policy 都有版本、摘要、来源和可测试完成条件。
7. 上游规则只能约束 Agent 行为，不能拥有流程控制权。

## 3.2 非目标

本期不实现：

- 新的 `/tdd`、`/review`、`/diagnose` 等公开命令；
- Superpowers 的完整插件安装；
- mattpocock/skills 的完整 Skill 安装；
- 独立 Issue Tracker；
- 自动创建 GitHub Issue；
- 独立任务清单系统；
- 新的用户 Router；
- 新的 worktree 生命周期；
- 后台异步 Agent 平台；
- 新的状态机主阶段。

# 4. 必须保持不变的公开接口

## 4.1 命令保持不变

```text
sdd init
sdd status
sdd new
sdd design
sdd plan
sdd build next
sdd build complete
sdd verify
sdd review
sdd archive
sdd auto
sdd codebase status
sdd codebase doctor
sdd codebase query
```

不得新增用户必须理解或执行的命令。

## 4.2 主状态保持不变

```text
NOT_INITIALIZED
INDEX_READY
SPEC_READY
DESIGN_READY
PLAN_READY
BUILD_WAITING_AGENT
BUILD_READY
VERIFY_READY
REVIEW_READY
ARCHIVED
```

现有运行态和错误态可以继续使用，但不得为上游 Skill 新增独立流程状态。

## 4.3 两种执行模式

### Auto 模式

```text
用户需求
  → sdd auto
  → LoopEngine 读取状态
  → 选择现有命令
  → 加载对应 Policy
  → Agent 执行 actionRequired
  → 提交结构化结果
  → LoopEngine 继续
  → ARCHIVED 或需要人工决策
```

### 手动模式

```text
用户指定现有命令
  → Core 执行当前阶段
  → 加载相同 Policy
  → 返回 CommandResult/actionRequired
  → 本次命令结束
```

两种模式的唯一区别是：是否自动执行下一阶段。

# 5. 总体架构

## 5.1 新增内部 Policy 层

```text
CLI
  ↓
Core / LoopEngine
  ↓ 选择阶段和 action
PhasePolicyRegistry
  ↓ 解析 policy IDs
PolicyCompiler
  ↓ 合并基础规则、阶段规则和运行上下文
Agent Adapter
  ↓ 适配 Claude/Codex/OpenCode/Generic
Agent
  ↓ 返回 artifact/result/evidence
Core Gate
```

## 5.2 依赖方向

允许：

```text
core → agent-policies
adapters → agent-policies
agent-policies → agent-protocol types
```

禁止：

```text
agent-policies → core command implementation
agent-policies → state store
agent-policies → git writer
agent-policies → worktree manager
```

Policy 层只能提供：

- Agent instructions；
- completion criteria；
- required evidence；
- policy metadata；
- Prompt 编译能力。

Policy 层不能执行状态流转。

# 7. Policy 数据模型

## 7.1 Policy 定义

```ts
export type PhasePolicyId =
  | "core-authority"
  | "security-boundaries"
  | "evidence-before-completion"
  | "bounded-clarification"
  | "spec-authoring"
  | "deep-module-design"
  | "design-it-twice"
  | "tracer-bullet-planning"
  | "expand-contract-migration"
  | "tdd-task-execution"
  | "context-pack-consumer"
  | "systematic-diagnosis"
  | "two-axis-review"
  | "handoff-and-traceability";

export interface PhasePolicyDefinition {
  id: PhasePolicyId;
  version: string;

  appliesTo: {
    commands?: string[];
    phases?: string[];
    actionTypes?: string[];
    failureCodes?: string[];
  };

  promptFile: string;
  requiredEvidence: string[];
  completionCriteria: string[];

  source?: {
    project: "sdd-harness" | "superpowers" | "mattpocock-skills";
    upstreamCommit?: string;
    adaptedFrom?: string[];
  };
}
```

## 7.2 Policy 引用

```ts
export interface PolicyRef {
  id: PhasePolicyId;
  version: string;
  digest: string;
}
```

## 7.3 Policy Bundle

```ts
export interface PolicyBundle {
  schemaVersion: "1.0.0";
  policies: PolicyRef[];
  instructions: string;
  requiredEvidence: string[];
  completionCriteria: string[];
}
```

## 7.4 AgentActionRequired 扩展

在不破坏旧字段的前提下增加可选字段：

```ts
export interface AgentActionRequired {
  type: string;

  policyBundle?: PolicyBundle;

  // 原有字段保持不变
  taskId?: string;
  changeId?: string;
  contextPack?: string;
  allowedFiles?: string[];
  forbiddenFiles?: string[];
  verification?: string[];
  resultFile?: string;
}
```

兼容要求：

- 老版本 Adapter 忽略 `policyBundle` 后仍可运行；
- 新版本 Adapter 优先使用 `policyBundle`；
- `policyBundle` 不得决定状态迁移；
- Core 仍以结构化 result 和真实 Git delta 为准。

# 9. 各阶段具体接入方案

## 9.1 `new`：有限需求澄清

借鉴 mattpocock 的 grilling，但不能原样使用持续追问模式。上游要求逐个解决设计树中的决策，并且能通过代码库找到的事实不应询问用户。

改造为 `bounded-clarification`：

### 行为规则

1. 先通过 codebase-memory 或 fallback scan 搜集事实。
2. 只有业务语义或不可推导决策才询问用户。
3. 一个 clarification item 只解决一个决策。
4. 每个问题必须提供：
   - 推荐答案；
   - 选择理由；
   - 备选项；
   - 对验收标准的影响。

5. 不得询问能够从代码、配置、Schema 或现有文档中找到的事实。
6. 所有会影响外部行为、数据模型、兼容性或验收标准的决策明确后停止澄清。

### Auto 行为

```text
存在未决业务问题
  → CLARIFYING
  → 等待用户
  → 用户回答
  → 自动继续 design、plan、build、verify、review、archive
```

### 手动行为

```text
sdd new
  → 返回 clarification
  → 用户补充后再次执行现有命令
```

不得增加 `sdd clarify`。

---

## 9.2 `new`：规格生成增强

只吸收 mattpocock `to-spec` 的内容结构，不采用其发布到 Issue Tracker 的流程。

建议规格必须覆盖：

- Problem Statement；
- Requirements；
- Scenarios；
- User-visible behavior；
- Testing decisions；
- Out of scope；
- Compatibility constraints；
- Failure behavior。

上游 `to-spec` 强调问题、用户故事、实现决策、测试决策和范围边界。

这些内容应映射到现有 OpenSpec 模型，不创建第二套 spec 格式。

---

## 9.3 `design`：Deep Module Design

吸收 mattpocock `codebase-design`：

- Module；
- Interface；
- Seam；
- Adapter；
- Depth；
- Leverage；
- Locality。

其核心原则是通过小接口隐藏更多行为，并让调用和测试使用同一个 seam。

### 设计产物必须回答

1. 本次变更涉及哪些 Module。
2. 每个 Module 的 Interface 是什么。
3. 测试通过哪个 Seam 验证。
4. 哪些复杂性被隐藏在 Module 内部。
5. 哪些 Adapter 已真实存在。
6. 是否引入了只有一个实现的假抽象。
7. 修改后的复杂性集中在哪里。
8. 关键错误模式、顺序约束和性能约束是什么。

### Design It Twice 条件触发

不新增命令，在复杂设计中由 Policy 条件触发。

建议条件：

```ts
interface DesignRisk {
  touchedPackages: number;
  publicInterfacesChanged: number;
  schemaChanged: boolean;
  persistenceChanged: boolean;
  externalProtocolChanged: boolean;
}
```

当满足任一条件：

- 修改公共协议；
- 修改持久化 Schema；
- 涉及三个以上 package；
- 新增跨层 Module；
- 存在不可逆架构决策；

则生成至少两个明显不同的设计方案，按以下维度比较：

- interface depth；
- locality；
- seam placement；
- migration cost；
- testability；
- compatibility。

上游的 `Design It Twice` 同样要求生成多个显著不同的接口方案并比较 depth、locality 和 seam placement。

---

## 9.4 `plan`：Tracer Bullet Planning

吸收 mattpocock `to-tickets` 的纵向切片原则，但任务仍写入现有 `tasks.json`，不创建外部 Ticket。

上游要求每个 ticket 是窄而完整的纵向路径、可单独验证，并明确 blocking edges。

### TaskDefinition 建议

```ts
export interface TaskDefinition {
  id: string;
  title: string;
  objective: string;

  userVisibleOutcome: string;
  acceptanceCriteria: string[];
  verification: string[];

  blockedBy: string[];
  allowedFiles: string[];
  forbiddenFiles?: string[];

  sliceType: "VERTICAL" | "EXPAND" | "MIGRATE" | "CONTRACT" | "REPAIR";

  contextPack: string;
}
```

### Planner 门禁

以下情况不得通过：

- 没有 `userVisibleOutcome`；
- 没有 verification；
- acceptance criteria 为空；
- `blockedBy` 指向不存在任务；
- 依赖图存在循环；
- 普通任务只修改一个技术层且不交付可验证行为；
- 任务明显超出一次 Agent 上下文；
- allowedFiles 是猜测而不是来自代码库分析。

### 大范围迁移

允许使用：

```text
EXPAND
  → MIGRATE-01
  → MIGRATE-02
  → ...
  → CONTRACT
```

适用于：

- Schema version；
- Agent protocol；
- 公共类型；
- 数据库字段；
- CLI 输出协议；
- 状态文件迁移。

---

## 9.5 `build`：TDD Task Execution

继续使用 Superpowers 已有 RED → GREEN → REFACTOR → VERIFY 模型，同时补充 mattpocock 对测试 seam 和纵向切片的要求。

Superpowers 把 TDD、verification-before-completion 和系统化调试作为核心工程方法。

mattpocock 的 TDD Skill 强调：

- 测试公开行为，不测试实现细节；
- 在稳定 seam 上验证；
- Red 必须先于 Green；
- 每次完成一个纵向切片。

### Build Policy

```text
1. 读取 AgentActionRequired。
2. 读取 Context Pack 和引用制品。
3. 确认 allowedFiles、forbiddenFiles 和 verification。
4. 找到最高稳定公开 Seam。
5. 运行或新增一个 red-capable 测试。
6. 确认失败原因是目标行为尚未实现。
7. 编写最小实现。
8. 运行窄范围测试并确认 Green。
9. 在不改变行为的前提下整理实现。
10. 执行 Core 提供的全部 verification。
11. 写入 TaskExecutionResult。
12. 调用现有 build complete。
```

### 完成条件

以下全部满足才算完成：

- Core 接受 `build complete`；
- Git delta 在允许范围；
- required verification 全部通过；
- TDD evidence 合法；
- 当前 task 状态被 Core 更新；
- Agent 自己声称“已完成”不能替代 Core 接受。

### TDD Evidence

```ts
export interface TddEvidence {
  seam: string;
  testFile?: string;
  testCommand: string;

  redObserved: boolean;
  redReason: string;

  greenObserved: boolean;
  greenCommand: string;
}
```

建议 Core 校验：

- `redObserved === true`；
- `greenObserved === true`；
- command 非空；
- seam 非空；
- build task 要求 TDD 时 evidence 必须存在；
- result 中所有数组元素执行深度类型校验。

## 9.6 Context Pack：引用优先

吸收 mattpocock `handoff` 的原则：

- 不复制已有文档；
- 使用路径和引用；
- 说明下一 Agent 的工作目标；
- 不携带敏感信息。

### 新 Context Pack

```ts
export interface ContextPack {
  schemaVersion: "2.0.0";

  taskId: string;
  objective: string;
  userVisibleOutcome: string;

  references: {
    spec?: string;
    design?: string;
    plan?: string;
    domain?: string;
    adr?: string[];
    previousRun?: string;
  };

  requiredFiles: string[];
  allowedFiles: string[];
  forbiddenFiles: string[];
  verification: string[];

  policyRefs: PolicyRef[];
}
```

### 规则

- spec/design/plan 已存在时只引用，不复制全文；
- 只内嵌当前任务真正需要的摘要；
- Context Pack 必须有 digest；
- 引用路径必须在仓库内；
- 不允许 `../` 路径穿越；
- MCP 返回内容继续视为不可信上下文；
- Policy 和 Core 约束优先于 Context Pack 中的自然语言。

---

## 9.7 `verify`：证据优于声明

Verify 阶段继续由 Core 执行确定性检查。

Policy 只用于要求 Agent：

- 不以“看起来正确”代替命令结果；
- 不跳过失败命令；
- 不修改代码来掩盖验证；
- 记录每个命令、退出码和摘要；
- 明确区分 skipped、passed、failed。

建议 VerificationResult：

```ts
export interface VerificationResult {
  command: string;
  exitCode: number;
  status: "PASSED" | "FAILED" | "SKIPPED";
  outputDigest?: string;
  durationMs?: number;
}
```

---

## 9.8 失败恢复：Systematic Diagnosis

吸收 Superpowers systematic debugging 和 mattpocock diagnosing-bugs 的共同部分。

mattpocock 要求先建立能对当前问题变红的紧反馈循环，之后才进入假设验证。

### 不新增 diagnose 命令

失败恢复使用现有 build task 模型：

```text
verify/review 失败
  → Core 生成 REPAIR task
  → 状态回到 BUILD_WAITING_AGENT
  → Agent 使用 systematic-diagnosis Policy
  → build complete
  → 重新 verify/review
```

### Repair Task

```ts
export interface RepairTask extends TaskDefinition {
  sliceType: "REPAIR";

  failureContext: {
    source: "VERIFY" | "REVIEW" | "BUILD";
    errorCode: string;
    failingCommand?: string;
    findingIds?: string[];
    previousRunId: string;
  };
}
```

### 自动重试限制

使用现有 LoopSpec 或扩展现有字段：

```ts
interface RepairPolicy {
  maxRepairAttemptsPerTask: number;
  maxRepeatedFailureSignature: number;
  stopOnScopeExpansion: boolean;
}
```

推荐默认：

```text
maxRepairAttemptsPerTask = 2
maxRepeatedFailureSignature = 2
stopOnScopeExpansion = true
```

### 必须暂停的条件

- 无法构造可重复反馈循环；
- 需要外部环境或权限；
- 修复要求修改 allowedFiles 之外的文件；
- 出现业务意义不同的多个修复方案；
- 同一失败签名达到限制；
- 修复会修改已确认需求；
- 安全或合规边界阻止继续。

---

## 9.9 `review`：双轴审查

吸收 mattpocock `code-review` 的双轴模型。

其核心是将 Standards 和 Spec 分开，防止一个维度掩盖另一个。

### ReviewReport

```ts
export interface ReviewReport {
  schemaVersion: "2.0.0";
  changeId: string;
  fixedPoint: string;

  standards: {
    status: "PASSED" | "FAILED";
    findings: ReviewFinding[];
  };

  spec: {
    status: "PASSED" | "FAILED" | "SKIPPED";
    findings: ReviewFinding[];
  };

  summary: {
    standardsFindingCount: number;
    specFindingCount: number;
  };
}
```

### Standards 轴

检查：

- 仓库现有规范；
- 模块边界；
- interface depth；
- seam；
- 重复代码；
- speculative abstraction；
- unrelated refactor；
- error handling；
- test quality；
- scope control。

### Spec 轴

检查：

- 漏实现；
- 错误实现；
- acceptance criteria 未覆盖；
- scope creep；
- 兼容性破坏；
- out-of-scope 行为；
- 测试与需求不一致。

### 门禁

```text
standards.status === PASSED
AND
spec.status === PASSED
```

任一失败：

- auto 模式生成 Repair Task；
- 手动模式返回失败及下一步 `build next`；
- 不进入 archive。

---

## 9.10 `archive`：Traceability 与 Handoff

Archive 不引入新流程，只增强追踪信息。

归档报告应记录：

- requirement → scenario；
- scenario → task；
- task → commit/delta；
- task → TDD evidence；
- task → verification；
- review finding → repair task；
- policy IDs、版本与 digest；
- upstream attribution；
- LoopRun ID。

# 10. Adapter 改造

## 10.1 当前问题

Claude、Codex、OpenCode manifest 中存在相同的：

- operating rules；
- Skill 内容；
- command template 文本。

这会造成：

- 三处修改；
- 不同 Adapter 漂移；
- Prompt token 浪费；
- 规则重要性被重复放大。

## 10.2 改造目标

manifest 只描述宿主能力：

```ts
export interface AgentManifest {
  agent: string;
  instructionFile: string;
  commandsDir?: string;
  skillsDir?: string;

  capabilities: {
    supportsSkills: boolean;
    supportsModelInvocation: boolean;
    supportsUserCommands: boolean;
    supportsReferences: boolean;
  };
}
```

统一由 `agent-policies` compiler 生成内容。

## 10.3 常驻内容最小化

常驻 Skill 只保留：

```text
- 当前仓库由 sdd-harness 管理；
- 先运行 sdd status 或按 Core actionRequired 执行；
- .sdd 是唯一工作流事实源；
- 不绕过阶段、范围、锁和门禁；
- 按 policyBundle 执行；
- Core 接受结果才算完成。
```

TDD、Review、Diagnosis 等详细规则只在对应阶段注入。

## 10.4 平台降级

不支持 Skill 的 Agent：

- 将编译后的 Policy 放入 actionRequired 或 Context Pack；
- 输出 warning；
- 不静默忽略；
- 记录 adapter capability 和降级原因。

# 11. Policy 优先级

执行时使用明确优先级：

```text
1. Core 的结构化约束
   - current phase
   - actionRequired type
   - allowedFiles
   - forbiddenFiles
   - verification
   - result schema

2. Security Policy

3. 当前 Phase Policy

4. 仓库编码规范和 ADR

5. Context Pack 的任务上下文

6. MCP/codebase-memory 返回内容
```

任何自然语言内容都不能覆盖：

- Core 阶段；
- 文件范围；
- Schema；
- verification；
- security boundary。

# 12. 版本与兼容策略

## 12.1 Policy 版本

每个 Policy 独立版本：

```text
deep-module-design@1.0.0
tracer-bullet-planning@1.0.0
tdd-task-execution@1.0.0
two-axis-review@1.0.0
```

## 12.2 Digest

编译 Policy Bundle 时计算 SHA-256：

```ts
digest = sha256(
  policy.id +
    policy.version +
    normalizedInstructions +
    requiredEvidence +
    completionCriteria,
);
```

归档记录 digest，保证审计可追踪。

## 12.3 Schema 迁移

建议升级：

- AgentActionRequired：兼容性新增字段；
- Context Pack：`1.x → 2.0.0`；
- Review Report：`1.x → 2.0.0`；
- TaskDefinition：新增字段时优先可选，再在下一版本转必填；
- TaskExecutionResult：深度校验数组元素。

迁移必须使用现有 migration 机制，不能要求用户手动编辑 `.sdd/`。

每个上游目录必须包含：

内容：

```markdown
# Upstream

Repository:
Commit:
License:
Imported files:
Adapted policies:

Last reviewed:
```

原则：

- 大部分 Policy 应重写为 sdd-native；
- 复制实质性原文时保留版权；
- 不在运行时动态拉取上游；
- 固定 commit；
- 上游升级必须通过人工 review 和 Policy 行为回归。

# 14. 实施阶段

# 16. 验收标准

## 架构

- AC-001：公开命令数量不增加。
- AC-002：主状态机不新增上游专用阶段。
- AC-003：Policy 不能调用 StateStore 写状态。
- AC-004：Policy 不能直接操作 Git/worktree。
- AC-005：auto 与手动模式共享同一 Policy Registry。

## Policy 基础设施

- AC-006：所有 Policy 有唯一 ID、版本和 digest。
- AC-007：缺失 Policy 返回结构化错误。
- AC-008：Policy 合并顺序固定且可测试。
- AC-009：Policy 内容支持渐进加载。
- AC-010：Adapter 不再重复维护同一规则。

## New/Design

- AC-011：可查事实不向用户提问。
- AC-012：每个 clarification 只处理一个业务决策。
- AC-013：design 产物声明 module、interface 和 seam。
- AC-014：高风险设计至少比较两个方案。
- AC-015：Policy 不引入额外设计确认节点。

## Plan

- AC-016：普通任务包含 userVisibleOutcome。
- AC-017：普通任务包含 verification。
- AC-018：依赖图无循环。
- AC-019：任务以纵向切片为默认。
- AC-020：大范围迁移支持 expand–migrate–contract。
- AC-021：不创建外部 Ticket。

## Build

- AC-022：Build actionRequired 包含 Policy Bundle。
- AC-023：Agent 必须读取 Context Pack。
- AC-024：modifiedFiles 等数组执行元素类型校验。
- AC-025：TDD 任务必须包含 red/green evidence。
- AC-026：Core 拒绝结果时状态不推进。
- AC-027：Agent 自称完成不能替代 build complete 成功。

## Failure Recovery

- AC-028：verify/review 可生成 REPAIR task。
- AC-029：Repair Task 仍通过现有 build next/complete 执行。
- AC-030：重复失败达到限制后暂停。
- AC-031：需要扩大范围时暂停并请求用户决策。
- AC-032：不存在无限自动重试。

## Review/Archive

- AC-033：review 分为 Standards 和 Spec 两轴。
- AC-034：任一轴失败都不能 archive。
- AC-035：review finding 可追踪到 Repair Task。
- AC-036：archive 记录 Policy ID、版本和 digest。
- AC-037：archive 记录 requirement → task → evidence 链路。

## 兼容与发布

- AC-038：旧 Adapter 能忽略新增可选字段。
- AC-039：现有项目自动迁移，无需手改 `.sdd/`。
- AC-040：所有 format、lint、typecheck、test 和 validation 通过。
- AC-041：auto/manual golden 语义等价。
- AC-042：未安装任何完整上游插件也能运行。

# 17. 风险与控制

| 风险              | 控制措施                                |
| ----------------- | --------------------------------------- |
| Prompt 过长       | 按阶段渐进加载                          |
| 上游规则抢占流程  | 不运行完整 Skill，只使用本地 Policy     |
| auto 行为变化过大 | 分阶段灰度接入                          |
| 手动与 auto 分叉  | 共用 registry 和 command implementation |
| 两套任务状态      | 只允许 `tasks.json` 和 Core 状态        |
| 无限失败修复      | retry budget + failure signature        |
| Adapter 规则漂移  | 统一 compiler                           |
| 上游更新破坏行为  | 固定 commit + golden eval               |
| Context Pack 过期 | 引用优先 + digest                       |
| 上游许可证遗漏    | vendor LICENSE + UPSTREAM.md            |

# 18. 推荐最终形态

```text
用户
 ├── auto：只输入需求
 └── manual：指定现有命令
          ↓
      sdd-harness Core
          ↓
  确定当前阶段和门禁
          ↓
  选择对应 Phase Policies
          ↓
    Adapter 编译并注入
          ↓
      Agent 执行任务
          ↓
  返回 artifact/result/evidence
          ↓
      Core 验证并推进
```

最终原则：

> Superpowers 提供已经验证过的执行纪律和 TDD 协议；mattpocock/skills 提供更细粒度的设计、切片、审查和 Prompt 组织方法；sdd-harness 始终保留唯一流程控制权。

上游内容只负责回答“当前阶段应该怎样高质量执行”，不能回答“下一阶段是什么”。
