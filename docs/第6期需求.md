# 2. 背景

sdd-harness 当前已经具备：

- 规格驱动的 Requirement/Scenario 模型；
- RED、GREEN、REFACTOR、VERIFY 任务链；
- Agent Action/Result 协议；
- 分阶段 Policy Bundle；
- Git delta、文件范围、命令和安全门禁；
- verify、review、archive 质量流程；
- Claude Code、Codex、OpenCode 等 Agent Adapter。

项目当前的 Policy 系统已经能够根据 command、phase、action type 和 failure code 渐进加载规则，并记录规则来源、版本和 digest。

当前确定性 review 主要覆盖：

- 需求和任务追踪；
- 文件范围；
- 未申报 Git 变化；
- 测试证据；
- 危险命令；
- 密钥泄漏。

但当前缺少专门针对以下问题的约束：

- Agent 重复实现代码库已有能力；
- 为单一实现创建接口、Factory 或额外层；
- 为少量逻辑引入新依赖；
- 引入没有明确需求的配置和扩展点；
- 修复症状而非共享根因；
- 代码变更规模不必要地扩大；
- 有意接受的简化没有留下升级触发条件。

Ponytail 的核心方法正好补充这一部分。它要求 Agent 在编码前依次判断：

1. 是否真的需要实现；
2. 代码库是否已有实现；
3. 标准库是否可以完成；
4. 平台原生能力是否可以完成；
5. 已安装依赖是否可以复用；
6. 最后才编写最小正确实现。

同时，它明确规定不能为了减少代码而牺牲安全、输入验证、数据完整性和错误处理。

# 3. 接入目标

本期接入应达到以下目标。

## 3.1 Agent 实施目标

Agent 在 design、plan、build 和 repair 阶段必须：

- 先检查现有代码，再决定新增实现；
- 优先修复共享根因，而非只修复一个调用路径；
- 优先复用现有模块、类型、工具函数和架构模式；
- 优先标准库、平台能力和已安装依赖；
- 避免没有第二个实际使用者的抽象层；
- 避免没有 Requirement 或 Design 依据的可配置性；
- 避免无计划新增依赖；
- 选择最小正确 diff，而不是最少字符；
- 保持现有安全、TDD、兼容性和恢复门禁。

## 3.2 Review 目标

`sdd review` 应增加简洁性审查能力：

- 识别未计划新增依赖；
- 记录变更规模指标；
- 识别有意接受的工程限制；
- 生成非阻断性的复杂度建议；
- 将确定性问题和主观建议分开；
- 不让 LLM 主观判断直接控制 Core 状态机。

## 3.3 Archive 目标

`sdd archive` 应保存：

- 变更代码规模；
- 新增和删除依赖；
- 简洁性 findings；
- 有意简化债务；
- Policy 来源和版本。

# 4. 非目标

本期明确不做以下事情。

## 4.1 不安装 Ponytail npm 包

不得将以下包作为 sdd-harness 运行时依赖：

```text
@dietrichgebert/ponytail
```

原因：

- sdd-harness 已有自己的 Policy Compiler；
- 已有自己的 Adapter；
- 已有自己的 MCP；
- 已有自己的状态机；
- 引入完整包会产生重复职责。

## 4.2 不引入 Ponytail Hooks

不得复制或安装 Ponytail 的：

- SessionStart Hook；
- UserPromptSubmit Hook；
- SubagentStart Hook；
- mode tracker；
- always-on prompt injection。

Ponytail 的 Hooks 会在每次会话和每个子 Agent 中持续注入规则。

这与 sdd-harness 的“只加载当前阶段 Policy Bundle”原则冲突。

## 4.3 不增加 CLI 命令

不得增加：

```text
sdd ponytail
sdd minimal
sdd audit
sdd debt
```

现有命令数量保持不变。

相关能力应进入：

- `sdd design`
- `sdd plan`
- `sdd build`
- `sdd review`
- `sdd archive`

## 4.4 不引入强度模式

不得增加：

```text
lite
full
ultra
off
```

sdd-harness 只采用一个固定的、平衡的 Minimality Policy。

## 4.5 不采用“最少测试”策略

不得因为 Ponytail 强调最小实现而减少：

- Requirement/Scenario 测试；
- RED/GREEN/REFACTOR/VERIFY；
- verification；
- 回归测试；
- 安全测试；
- 迁移测试。

最小化只针对生产代码和无必要结构，不针对质量证据。

# 5. 核心设计原则

## 5.1 优先级规则

所有 Agent Policy 必须遵循以下优先级：

```text
Core 状态机
→ Spec 和 Acceptance Criteria
→ Security Boundary
→ Data Integrity
→ Compatibility / Migration
→ TDD 和 Verification
→ Minimality
→ 输出简洁性
```

当 Minimality Policy 与高优先级规则冲突时，Minimality 必须让步。

例如：

- Requirement 明确要求缓存，则不能以 YAGNI 为由拒绝；
- 数据迁移需要兼容层，则不能因为只有一个实现而删除接口；
- 安全边界要求额外验证，则不能以减少代码为由省略；
- VERIFY 要求完整测试，则不能只保留一个 smoke test。

## 5.2 Core 继续作为唯一事实源

Ponytail-derived Policy 只能影响 Agent 如何实施，不得改变：

- 当前阶段；
- Task 状态；
- Git delta；
- verification 结果；
- review 阻断结果；
- archive 资格。

Core CommandResult 和 `.sdd` 制品仍是唯一事实源。

## 5.3 主观判断不得成为硬门禁

以下问题通常带有主观性：

- 类是否太多；
- 抽象是否过度；
- 文件是否太长；
- 是否可以少写十行；
- 模块是否“看起来复杂”。

这类 finding 默认只能是：

```text
MINOR
INFO
```

不得直接阻断归档。

只有可确定性证明的问题才能成为 MAJOR。

# 6. 总体架构

接入后的处理流程如下：

```text
Requirement / Spec
        │
        ▼
Policy Resolver
        │
        ├── existing phase policies
        └── Ponytail-derived minimality policies
        │
        ▼
Agent Policy Bundle
        │
        ▼
Design / Plan / Build Agent
        │
        ▼
Git delta + task result
        │
        ▼
Deterministic Review
        │
        ├── existing security/spec/test gates
        ├── dependency change gate
        ├── complexity metrics
        └── deliberate debt scanner
        │
        ▼
Review Report
        │
        ▼
Archive
```

Ponytail-derived Policy 不需要新的进程、插件、Hook 或 MCP。

# 8. Policy 文件

## 8.1 minimal-implementation

新增：

```text
packages/agent-policies/policies/shared/minimal-implementation.md
```

建议内容：

```markdown
# Minimal Implementation

在满足 Spec、Acceptance Criteria、安全、兼容性、迁移和测试要求的前提下，选择最小正确实现。

实施前按顺序检查：

1. 当前 Requirement 是否真的要求新增实现。
2. 代码库中是否已经存在可复用的 helper、type、module、adapter 或模式。
3. Node.js 标准库、语言标准库或数据库原生能力是否已经覆盖。
4. 当前平台原生能力是否已经覆盖。
5. 已安装依赖是否已经覆盖。
6. 只有以上方案均不满足时，才新增实现。

规则：

- 明确 Requirement 不得因“更简单”而删除或弱化。
- 修改共享根因，不只补一个调用路径。
- 不创建只有一个真实实现的接口、Factory 或注册层，除非 Design 明确要求。
- 不添加没有真实使用者的扩展点、配置项、Feature Flag 或 Plugin API。
- 不为少量、稳定逻辑新增依赖。
- 优先复用项目现有约定，不创建平行实现。
- 选择最小正确 diff，而不是最少字符。
- 不得省略输入验证、错误恢复、安全检查、数据完整性、兼容逻辑、迁移逻辑、可访问性和测试证据。
- 若有意接受一个有边界的简化，在代码附近记录：
  `sdd-debt: <限制>; trigger=<升级触发条件>; upgrade=<升级方案>`
```

## 8.2 root-cause-minimal-fix

新增：

```text
packages/agent-policies/policies/failure/root-cause-minimal-fix.md
```

建议内容：

```markdown
# Root Cause Minimal Fix

修复失败时，不要只修复错误消息描述的单一路径。

执行步骤：

1. 复现失败并记录 failure signature。
2. 找到真正负责该行为的共享函数、状态迁移或协议边界。
3. 检查所有调用者和相邻路径。
4. 优先在最靠近根因的位置修复一次。
5. 避免在多个调用点复制相同 guard。
6. 添加能在根因再次出现时失败的最小完整回归测试。
7. 不扩大到与当前 failure 无关的重构。
8. 不因减少 diff 而保留同类已知错误。
```

该 Policy 与现有 `systematic-diagnosis` 一起使用，不替代它。

# 10. Agent Evidence 扩展

Minimality Policy 不应依赖自由文本声明。Agent handoff 应增加结构化 evidence。

在 Agent Task Result 或 Policy Evidence 中增加：

```ts
export interface MinimalityEvidence {
  reusedExisting: string[];
  standardLibraryChoices: string[];
  nativePlatformChoices: string[];
  dependenciesAdded: DependencyDecision[];
  abstractionsAdded: AbstractionDecision[];
  deliberateDebts: DeliberateDebtDeclaration[];
}

export interface DependencyDecision {
  name: string;
  manifest: string;
  reason: string;
  requiredBy: string[];
}

export interface AbstractionDecision {
  name: string;
  file: string;
  consumers: string[];
  reason: string;
}

export interface DeliberateDebtDeclaration {
  file: string;
  ceiling: string;
  trigger: string;
  upgrade: string;
}
```

字段可以先作为可选字段加入，避免破坏旧 Agent Adapter。

Core 不得只依赖 Agent evidence 判断依赖是否真的新增。依赖变化仍要通过 Git diff 和 manifest 解析确定。

## 11.1 ChangeComplexityMetrics

定义：

```ts
export interface ChangeComplexityMetrics {
  filesAdded: number;
  filesModified: number;
  filesDeleted: number;
  linesAdded: number | null;
  linesDeleted: number | null;
  netLines: number | null;
  dependenciesAdded: number;
  dependenciesRemoved: number;
  deliberateDebtCount: number;
}
```

说明：

- 文件指标来自 Git delta；
- 行数指标通过 `git diff --numstat` 获取；
- Git 不可用时行数为 `null`；
- 指标只用于报告，不设置绝对失败阈值。

禁止实现：

```text
改动超过 300 行就失败
文件超过 5 个就失败
```

## 11.2 Dependency Delta

新增：

```text
packages/core/src/quality/dependency-delta.ts
```

第一期必须确定性支持：

```text
package.json
```

比较以下字段：

```text
dependencies
devDependencies
peerDependencies
optionalDependencies
```

定义：

```ts
export interface DependencyDelta {
  manifest: string;
  section:
    | "dependencies"
    | "devDependencies"
    | "peerDependencies"
    | "optionalDependencies";
  name: string;
  before: string | null;
  after: string | null;
  change: "ADDED" | "REMOVED" | "UPDATED";
}
```

处理规则：

### 新增依赖

若 Agent 新增依赖，但 plan/design 中没有对应 dependency decision：

```text
category = UNPLANNED_DEPENDENCY
severity = MAJOR
axis = STANDARDS
```

返回错误码：

```text
E_UNPLANNED_DEPENDENCY
```

### 升级依赖版本

默认生成 MINOR finding。

以下情况可升级为 MAJOR：

- 主版本升级；
- 新增安全或运行时权限；
- 与 Requirement 无关；
- 没有迁移说明。

### 删除依赖

记录为 INFO。

若删除后仍存在 import 或 require，应由 build/typecheck/verify 检测，不在本模块重复实现。

---

## 11.3 Planned Dependency 数据来源

Plan artifact 应支持：

```ts
export interface PlannedDependency {
  name: string;
  manifest: string;
  action: "ADD" | "UPDATE" | "REMOVE";
  reason: string;
  requirementIds: string[];
}
```

该字段为可选：

```ts
dependencies?: PlannedDependency[];
```

旧 plan 没有该字段时：

- 没有依赖变化：正常；
- 存在新增依赖：阻断，并要求重新执行 plan 或创建 repair task。

# 12. Deliberate Debt

## 12.1 代码标记格式

统一使用：

```text
sdd-debt: <限制>; trigger=<升级触发条件>; upgrade=<升级方案>
```

示例：

```ts
// sdd-debt: 当前使用全局锁; trigger=并发等待超过5%; upgrade=改为按项目分片锁
```

不使用：

```text
ponytail:
TODO:
later:
temporary:
```

原因是这些标记不具备稳定结构。

## 12.2 解析器

新增：

```text
packages/core/src/quality/deliberate-debt.ts
```

定义：

```ts
export interface DeliberateDebt {
  file: string;
  line: number;
  ceiling: string;
  trigger: string;
  upgrade: string;
}
```

扫描范围：

- 当前 Git delta 中新增或修改的文本文件；
- 排除 `.git`、`node_modules`、构建目录和二进制文件；
- 限制单文件大小；
- 不扫描整个仓库，避免每次 review 成本扩大。

解析失败时生成：

```text
category = DELIBERATE_DEBT
```

# 16. Advisor 与确定性门禁的边界

Ponytail 的 review 原本只关注过度设计，并明确排除正确性、安全和性能。

在 sdd-harness 中应按以下方式拆分。

## 16.1 Core 确定性能力

Core 可以可靠判断：

- 是否新增依赖；
- 新增依赖是否在计划中；
- Git delta 规模；
- 是否存在 `sdd-debt`；
- debt 是否缺少 trigger；
- 文件是否超出任务范围；
- 是否有未申报改动。

## 16.2 Agent Advisor 能力

Agent 可以给出建议：

- 可以删除的 wrapper；
- 可以使用标准库替代的代码；
- 单实现接口是否有必要；
- 是否重复已有 helper；
- 是否存在无消费者扩展点；
- 是否有过度通用的配置。

这些 finding 必须标记：

```text
deterministic = false
```

不得由 Agent 自己直接将 review 结果设置为 BLOCK。
